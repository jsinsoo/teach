과학고 정보 수업: 목표지향적 개념기반 수업지도안

수업 주제: III. 알고리즘과 프로그래밍 - 321. 문제 분해: 복잡성의 지도를 그리는 기술

수업 시작 멘트

인류의 가장 위대한 공학적 성취 중 하나인 아폴로 계획을 생각해봅시다. 그 목표는 단 하나, "인간을 달에 보내고 무사히 귀환시킨다"였습니다. 하지만 NASA의 과학자들은 이 거대한 목표를 '로켓 발사', '지구 궤도 진입', '달 착륙', '월면 활동', '지구 귀환' 등 수백, 수천 개의 작고 관리 가능한 문제들로 나누었습니다. 오늘 우리는 이 위대한 공학자들처럼, 아무리 거대하고 복잡해 보이는 문제라도 잘게 나누어 정복하는 가장 근본적인 문제 해결의 기술, '문제 분해'를 배우겠습니다.

1. 학습 목표

개념적 목표

  • 문제 분해(Decomposition)가 복잡한 문제를 해결 가능한 작은 하위 문제들의 집합으로 나누는 핵심적인 문제 해결 전략임을 설명할 수 있다.
  • 하향식 설계(Top-down Design)의 개념을 이해하고, 전체 문제에서 시작하여 세부 기능으로 구체화하는 과정을 설명할 수 있다.
  • 추상화(Abstraction)가 문제 분해 과정에서 불필요한 세부사항을 숨기고 각 단위의 역할에 집중하게 하는 원리임을 이해한다.

절차적 목표

  • 주어진 복잡한 문제의 요구사항을 분석하여, 독립적인 기능을 수행하는 여러 개의 하위 문제로 분해할 수 있다.
  • 분해된 각 하위 문제를 해결하는 개별 함수를 정의하고, 각 함수의 입력과 출력을 설계할 수 있다.
  • 설계된 여러 함수들을 조합하여 전체 문제를 해결하는 프로그램을 구현할 수 있다.

태도적 목표

  • 크고 막막한 문제를 만났을 때, 두려워하지 않고 체계적으로 분석하고 분해하려는 공학적 접근 방식을 내재화한다.
  • 코딩을 시작하기 전에 충분한 설계 과정을 거치는 것의 중요성을 인지하고, 계획적으로 프로그래밍하는 습관을 기른다.

2. 핵심 개념 상세 설명

문제 분해 (Problem Decomposition)

"코끼리를 먹는 유일한 방법은 한 번에 한 입씩 먹는 것이다"라는 격언처럼, 거대하고 복잡한 문제를 한 번에 해결하려 하지 않고, 더 작고, 관리하기 쉬우며, 독립적으로 해결할 수 있는 여러 개의 하위 문제로 나누는 과정입니다. 이는 소프트웨어 공학의 가장 근본적인 원리 중 하나이며, '분할 정복(Divide and Conquer)' 전략의 기초가 됩니다. 잘 분해된 문제는 각 부분을 나누어 팀원들과 협력하여 해결할 수도 있습니다.

하향식 설계 (Top-down Design)

문제 분해를 수행하는 대표적인 방법입니다. 가장 크고 추상적인 문제(Top)에서 시작하여, 이 문제를 해결하기 위해 필요한 주요 기능들로 나누고, 다시 그 각 기능을 더 세부적인 기능으로 나누어 내려가는(down) 방식입니다. 마치 회사의 조직도를 그리는 것처럼, 'CEO(전체 프로그램) → 각 부서(핵심 기능 모듈) → 각 팀(세부 기능 함수)'의 형태로 구조를 설계합니다.

추상화 (Abstraction)

문제 분해 과정에서 각 하위 문제의 '무엇(What)'을 할지에만 집중하고, '어떻게(How)' 할지에 대한 복잡한 세부 내용은 숨기는 것을 의미합니다. 우리는 자동차를 운전할 때, 엔진 내부의 복잡한 폭발 행정을 알 필요 없이 '가속 페달을 밟는다'는 간단한 인터페이스만 사용합니다. 프로그래밍에서는 '함수'가 추상화의 가장 대표적인 도구입니다. 함수의 사용자는 함수 이름과 입력, 출력만 알면 될 뿐, 그 내부가 어떻게 구현되었는지는 몰라도 함수를 사용할 수 있습니다.

모듈성 (Modularity)

문제 분해를 통해 만들어진 각 기능 단위(함수, 클래스, 파일)가 독립적으로 작동하고, 다른 부분에 미치는 영향을 최소화하는 특성을 의미합니다. 모듈성이 높은 프로그램은 부품 교체가 쉬운 기계처럼, 특정 기능의 수정이나 개선이 다른 부분에 예기치 않은 오류를 일으킬 가능성이 적어 유지보수가 매우 용이합니다.

3. 수업의 흐름 상세 기술

1단계: 도입 (5분) - 은행 ATM 기기 분해하기

"은행 ATM 기기는 매우 복잡한 기계처럼 보입니다. 하지만 그 본질을 들여다보면 어떨까요? '입금하기', '출금하기', '계좌 조회하기', '송금하기'라는 몇 가지 핵심적인 기능들의 조합일 뿐입니다. 만약 우리가 ATM 기계를 만드는 엔지니어라면, 'ATM 만들기'라는 거대한 하나의 목표 대신, 이 각각의 기능을 따로따로 개발하여 나중에 조립하는 것이 훨씬 현명할 것입니다. 이것이 바로 문제 분해의 힘입니다. 오늘 우리는 이 문제 분해를 통해 복잡한 소프트웨어를 설계하는 건축가가 되어 보겠습니다."

2단계: 전개 (35분) - "학생 성적 처리 프로그램" 설계 및 구현

활동 1: 1단계 - 문제 이해 및 분해 (하향식 설계)

문제 제시: "학생들의 이름과 여러 과목의 점수가 담긴 데이터를 처리하여, 각 학생의 총점과 평균을 계산하고, 평균 점수에 따라 등급을 부여하여 최종 성적표를 출력하는 프로그램을 만드시오."

학생들과 함께 이 문제를 가장 큰 기능 단위로 분해합니다.

  • 가장 큰 문제: 학생 성적 처리 프로그램
    • 하위 문제 1: 데이터를 프로그램으로 읽어오기
    • 하위 문제 2: 각 학생별로 성적 처리하기
      • 더 작은 문제 2-1: 한 학생의 총점 계산하기
      • 더 작은 문제 2-2: 한 학생의 평균 계산하기
      • 더 작은 문제 2-3: 평균 점수로 등급 결정하기
    • 하위 문제 3: 처리된 결과를 형식에 맞게 출력하기

"이처럼 거대한 문제가 여러 개의 해결 가능한 작은 문제들로 나뉘었습니다. 이제 각 문제를 하나의 '함수'로 만들어 봅시다."

활동 2: 2단계 - 해결 전략 수립 (의사코드와 함수 설계)

수업 마무리 멘트

고대 로마인들은 "Divide et Impera(분할하여 통치하라)"라는 전략으로 거대한 제국을 다스렸습니다. 거대한 적을 한 번에 상대하는 대신, 여러 개의 작은 세력으로 나누어 각개 격파하는 것이죠. 프로그래밍에서 '문제 분해'는 바로 이 로마인의 지혜와 같습니다. 여러분 앞에 어떤 거대하고 복잡한 문제가 나타나더라도, 당황하지 마십시오. 침착하게 그 문제를 분석하고, 작은 조각으로 나누어 하나씩 정복해 나가면 됩니다. 오늘 배운 이 '분할 정복'의 전략으로, 여러분이 마주할 모든 도전을 지배하는 현명한 문제 해결사가 되기를 바랍니다.

4. 진단 및 형성평가

진단평가 (OX 퀴즈 10개)

1. 문제 분해는 큰 문제를 작고 관리 가능한 하위 문제로 나누는 과정이다. (O)

풀이: 이것이 문제 분해(Decomposition)의 정확한 정의이며, 복잡한 문제를 해결하는 첫걸음입니다.

2. 알고리즘 설계 시, 문제 설명에 나오는 '동사'는 주로 클래스의 '속성'이 된다. (X)

풀이: 동사(Verbs)는 '행동'을 나타내므로, 클래스의 '메서드(method)'가 되는 경우가 일반적입니다. '명사(Nouns)'가 클래스나 속성이 됩니다.

3. 추상화(Abstraction)는 객체의 모든 세부적인 내부 구현을 외부에 공개하는 것을 의미한다. (X)

풀이: 추상화는 그 반대입니다. 복잡한 내부 구현은 '숨기고', 사용자가 사용하기 쉬운 간단한 인터페이스(메서드 등)만 외부에 공개하는 것을 의미합니다.

4. 의사코드(Pseudocode)는 실제 실행은 불가능하지만, 알고리즘의 논리를 설계하고 표현하는 데 유용하다. (O)

풀이: 의사코드는 특정 언어의 문법에 얽매이지 않고, 아이디어를 빠르게 스케치하고 논리 흐름을 검증하는 데 매우 효과적인 도구입니다.

5. "is-a" 관계(예: 강아지는 동물이다)는 상속으로, "has-a" 관계(예: 자동차는 엔진을 가진다)는 포함으로 모델링하는 것이 일반적이다. (O)

풀이: 이는 객체지향 설계의 중요한 원칙입니다. `class Dog(Animal)`은 'is-a', `self.engine = Engine()`은 'has-a' 관계를 나타냅니다.

6. 하향식(Top-down) 설계는 가장 작고 기본적인 부품부터 만든 뒤, 이를 조립하여 전체 시스템을 완성하는 방식이다. (X)

풀이: 이는 상향식(Bottom-up) 설계에 대한 설명입니다. 하향식 설계는 전체 시스템에서 시작하여 점차 세부적인 부분으로 나누어 내려가는 방식입니다.

7. 모델링은 현실 세계의 대상을 컴퓨터가 이해할 수 있도록 단순화하고 구조화하는 과정이다. (O)

풀이: 모델링의 핵심은 현실의 모든 복잡성을 그대로 옮기는 것이 아니라, 문제 해결에 '필요한' 핵심적인 속성과 행동만을 추려내어 표현하는 것입니다.

8. 좋은 설계 없이 코딩을 시작하면, 나중에 코드를 수정하고 확장하기 더 어려워질 수 있다. (O)

풀이: 설계 없는 코드는 구조가 뒤죽박죽이 되기 쉽습니다. 이는 '기술 부채(technical debt)'를 만들어, 당장은 빠를지 몰라도 장기적으로는 유지보수 비용을 크게 증가시킵니다.

9. C++의 헤더 파일(.h)에 클래스의 구조를 선언하는 것은, 파이썬에서 pass를 사용해 스켈레톤 코드를 만드는 것과 철학적으로 유사하다. (O)

풀이: 두 방식 모두 '설계(인터페이스)'와 '구현(내용)'을 분리하여 생각하고, 전체적인 구조를 먼저 잡는다는 공통된 철학을 가지고 있습니다.

10. 객체지향 분석/설계는 오직 객체지향 언어(파이썬, C++ 등)에서만 유용한 사고방식이다. (X)

class Clock:
    def __init__(self, hour, minute, second):
        self.hour = hour
        self.minute = minute
        self.second = second
    
    def __str__(self):
        h = str(self.hour).zfill(2)
        m = str(self.minute).zfill(2)

        return f"{h}:{m}:{s}"

class Clock:

문제 5: 위 Student 클래스에, 모든 수강 과목의 평균 점수를 계산하는 `get_average_score()` 메서드를 추가하세요.

# 문제 4의 클래스 정의에 이어서...
class Student:
    # ... __init__, add_subject ...
    def get_average_score(self):
        if not self.subjects:
            return 0.0
        
        total_score = sum([subj.score for subj in self.subjects])
        return total_score / len(self.subjects)

s1 = Student("뉴턴")
s1.add_subject("고전역학", 100)
s1.add_subject("광학", 95)
print(f"{s1.name}의 평균 점수: {s1.get_average_score()}") # 97.5

문제 7: 중력이 있는 '자유 낙하 물체'를 모델링하세요. `FallingObject` 클래스는 `mass`, `position`, `velocity`를 속성으로 가집니다. 1초의 시간이 흐르는 것을 시뮬레이션하는 `step()` 메서드를 구현하세요. (단, 중력가속도 g=9.8, 가속도 a = g)

class FallingObject:
    def __init__(self, mass, initial_position=1000.0):
        self.mass = mass
        self.position = initial_position
        self.velocity = 0.0 # 초기 속도는 0
        self.g = 9.8

    def step(self, time_delta=1.0):
        # v = v0 + at
        self.velocity += self.g * time_delta
        # s = s0 - v*t (아래로 떨어지므로)
        self.position -= self.velocity * time_delta
        if self.position < 0:
            self.position = 0

    def __str__(self):
        return f"위치: {self.position:.2f}m, 속도: {self.velocity:.2f}m/s"

ball = FallingObject(mass=10)
print(f"t=0s, {ball}")
ball.step()
print(f"t=1s, {ball}")
ball.step()

문제 8: 간단한 'DNA 서열' 클래스를 만드세요. `__init__`에서 서열 문자열을 받고, 'GC 함량'(전체 서열 중 G와 C의 비율)을 계산하여 반환하는 `gc_content()` 메서드를 구현하세요.

class DNASequence:
    def __init__(self, sequence):
        # 입력된 서열을 대문자로 통일하여 저장
        self.sequence = sequence.upper()

    def gc_content(self):
        if not self.sequence:
            return 0.0
        g_count = self.sequence.count('G')
        c_count = self.sequence.count('C')
        return (g_count + c_count) / len(self.sequence) * 100

my_dna = DNASequence("atgcnnatgc")
print(f"'{my_dna.sequence}'의 GC 함량: {my_dna.gc_content():.2f}%")
상세한 풀이와 설명: 생물정보학의 기본 개념을 클래스로 모델링합니다. DNA 서열이라는 데이터를 `sequence` 속성에 저장하고, GC 함량 계산이라는 분석 기능을 `gc_content` 메서드로 제공합니다. 문자열의 `.count()` 메서드를 효과적으로 활용합니다.
