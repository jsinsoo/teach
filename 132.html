1. 학습 목표

개념적 목표

  • IoT 시스템 개발의 전체 프로세스(문제 정의 → 요구사항 분석 → 아키텍처 설계 → 구현 및 테스트)를 설명할 수 있다.
  • 주어진 문제 상황을 해결하기 위해 필요한 센서, 액추에이터, 통신 방식, 플랫폼 기능을 스스로 정의할 수 있다.
  • 하드웨어(임베디드 시스템), 소프트웨어(펌웨어, 플랫폼, 앱)가 어떻게 데이터를 주고받으며 유기적으로 상호작용하는지 시스템 전체의 관점에서 설명할 수 있다.
  • 프로토타이핑(Prototyping)의 개념을 이해하고, 시뮬레이션을 통해 실제 하드웨어 없이 시스템의 핵심 로직을 검증하는 것의 중요성을 설명할 수 있다.

절차적 목표

  • 특정 문제(예: 스마트 조명 시스템)를 해결하기 위한 IoT 시스템의 요구사항 명세를 작성할 수 있다.
  • 시스템의 각 구성 요소(디바이스, 플랫폼, 앱)를 Python 클래스로 추상화하고 모델링할 수 있다.
  • 클래스 간의 메서드 호출을 통해 디바이스, 플랫폼, 앱의 상호작용 및 데이터 흐름을 코드로 구현할 수 있다.
  • 단순한 단일 기능이 아닌, 여러 조건이 결합된 복합적인 규칙(Rule)을 코드로 구현하고 테스트할 수 있다.

태도적 목표

  • 추상적인 아이디어를 구체적인 시스템 설계도와 코드로 변환하는 과정에서 논리적 사고력과 문제 해결 능력을 기른다.
  • 하나의 정답이 아닌, 다양한 설계와 구현 방식을 탐색하며 창의적이고 유연한 사고방식을 함양한다.
  • 시스템 설계 시 발생할 수 있는 예외 상황(네트워크 오류, 센서 오류 등)을 미리 예측하고, 이를 대비하는 안정적인 코드를 작성하려는 엔지니어링 마인드를 기른다.

2. 핵심 개념 상세 설명

IoT 시스템 설계 프로세스

아이디어 구상부터 실제 구현까지의 체계적인 단계입니다.

  • 문제 정의 (Problem Definition): 해결하고자 하는 문제가 무엇인지, 이 시스템이 왜 필요한지를 명확히 정의합니다. (예: "밤에 공부할 때 일일이 조명을 켜고 끄기 불편하고, 자리를 비웠을 때 낭비되는 전력이 아깝다.")
  • 요구사항 분석 (Requirements Analysis): 시스템이 반드시 갖춰야 할 기능(Functional)과 성능/보안 등의 비기능적(Non-functional) 요구사항을 상세히 정의합니다. (예: "어두워지면 켜져야 한다", "사람이 없으면 꺼져야 한다", "스마트폰으로 원격 제어가 가능해야 한다.")
  • 아키텍처 설계 (Architecture Design): 시스템을 주요 구성 요소(하드웨어, 소프트웨어, 네트워크 등)로 나누고, 이들 간의 역할과 데이터 흐름, 통신 방식을 설계합니다. 어떤 센서와 액추에이터를 쓸지, 통신은 Wi-Fi로 할지, 플랫폼은 어떤 기능을 가질지 등을 결정합니다.
  • 구현 (Implementation): 설계에 따라 각 구성 요소의 하드웨어를 구성하고 소프트웨어(펌웨어, 플랫폼, 앱)를 코드로 작성합니다.
  • 테스트 및 배포 (Test & Deployment): 구현된 시스템이 요구사항대로 정확히 동작하는지 다양한 시나리오에 따라 테스트하고, 오류를 수정하여 실제 환경에 배포합니다.

프로토타이핑 (Prototyping)

실제 제품을 만들기 전에, 핵심 기능만을 간단하게 구현하여 아이디어를 검증하고 문제점을 미리 파악하는 과정입니다. 소프트웨어 시뮬레이션은 실제 하드웨어(MCU, 센서) 없이도 Python과 같은 범용 언어를 사용하여 시스템의 전체적인 로직과 데이터 흐름을 검증할 수 있는 매우 효과적인 프로토타이핑 방법입니다.

상태 기계 (State Machine)

시스템이 가질 수 있는 여러 '상태(State)'와, 특정 '이벤트(Event)'에 의해 한 상태에서 다른 상태로 '전이(Transition)'되는 것을 모델링하는 방식입니다. (예: 조명 시스템은 'ON', 'OFF', 'AUTO' 세 가지 상태를 가질 수 있으며, '사람 감지'라는 이벤트에 의해 'OFF' 상태에서 'ON' 상태로 전이될 수 있습니다.) 복잡한 시스템의 동작을 명확하고 체계적으로 설계하는 데 도움이 됩니다.

Stage 2: 시스템 설계하기 (아키텍처 설계) (10분)

칠판에 시스템 아키텍처를 함께 그려봅니다.

  • Device (스마트 스탠드): MCU, 조도 센서, PIR(인체 감지) 센서, LED(액추에이터), Wi-Fi 통신 모듈.
  • Platform (클라우드): 디바이스 관리, 데이터 수집, 규칙 엔진, API 서버.
  • Application (스마트폰 앱): 상태 조회 및 제어 명령 전송.
  • 데이터 흐름 정의:
    • (상향) 스탠드가 주기적으로 센서 값과 현재 상태를 플랫폼에 전송한다.
    • (하향) 앱이 플랫폼의 API를 호출하면, 플랫폼이 스탠드에 제어 명령을 전송한다.

Stage 3: 코드로 생명 불어넣기 (구현 및 통합) (15분)

  • 펌웨어 구현: SmartLampDevice 클래스 내부를 채워봅니다. 센서 값을 읽고, 'AUTO' 모드일 때의 자동 점등/소등 로직을 구현합니다.
  • 플랫폼 구현: CloudPlatform 클래스 내부를 채웁니다. 디바이스의 데이터를 받아 저장하고, 앱의 명령을 받아 해당 디바이스에 전달하는 로직을 구현합니다.
  • 통합 시뮬레이션 코드: 위에서 설계한 클래스들을 실제로 객체로 만들고, 서로의 메서드를 호출하며 전체 시스템이 동작하는 과정을 시뮬레이션합니다.
import time

# 간단화된 통합 시스템 구현
class CloudPlatform:
    def __init__(self):
        self.device_handlers = {} # 디바이스 제어를 위한 핸들러 저장
        self.device_latest_status = {}

    def register_device(self, device_id, device_handler):
        self.device_handlers[device_id] = device_handler

    def handle_device_data(self, device_id, data):
        print(f"PLATFORM: '{device_id}' 데이터 수신 -> {data}")
        self.device_latest_status[device_id] = data

    def handle_app_command(self, device_id, command):
        print(f"PLATFORM: 앱으로부터 '{command}' 명령 수신 -> '{device_id}'에 전달")
        if device_id in self.device_handlers:
            handler = self.device_handlers[device_id]
            if command == 'TURN_ON': handler.turn_on()
            elif command == 'TURN_OFF': handler.turn_off()

class SmartLampDevice:
    def __init__(self, device_id, platform):
        self.id = device_id
        self.platform = platform
        self.is_on = False
        self.platform.register_device(self.id, self) # 플랫폼에 자신을 등록

    def turn_on(self): self.is_on = True; print(f"LAMP({self.id}): 켜짐.")
    def turn_off(self): self.is_on = False; print(f"LAMP({self.id}): 꺼짐.")

    def run_loop_once(self, light_level, motion_detected):
        # 센서 데이터와 현재 상태를 플랫폼에 보고
        status_data = {"light": light_level, "motion": motion_detected, "is_on": self.is_on}
        self.platform.handle_device_data(self.id, status_data)

# 시스템 실행
print("\n2. [구현 및 통합] 시스템의 각 부분을 코드로 구현하고 연결합니다.")
platform = CloudPlatform()
lamp = SmartLampDevice("desk_lamp_01", platform)

print("\n--- 시나리오 1: 자동 점등 (펌웨어 로직) ---")
lamp.run_loop_once(light_level=20, motion_detected=True) # 어둡고 움직임 있음
if 20 < 50 and True: lamp.turn_on() # 펌웨어의 자체 판단 로직

print("\n--- 시나리오 2: 앱으로 원격 제어 (플랫폼 로직) ---")
platform.handle_app_command("desk_lamp_01", "TURN_OFF")
lamp.run_loop_once(light_level=20, motion_detected=True) # 앱 명령 후 상태 보고

상세 설명: 이 코드는 설계된 아키텍처가 실제로 어떻게 동작하는지 보여주는 통합 시뮬레이션입니다. register_device: 디바이스가 부팅될 때, 자신의 제어 핸들러(자기 자신, self)를 플랫폼에 등록하는 과정입니다. 시나리오 1: 디바이스(lamp)가 센서 값을 읽어 플랫폼에 보고(handle_device_data)하고, 동시에 펌웨어 내부 로직(if 20 < 50 and True: ...)에 따라 스스로 불을 켭니다. 시나리오 2: 사용자 앱이 플랫폼을 통해(handle_app_command) 원격으로 끄기 명령을 내립니다. 플랫폼은 등록된 핸들러를 찾아 lamp.turn_off()를 호출하여 기기를 제어합니다. 이는 IoT 시스템의 상향링크(데이터 보고)와 하향링크(명령 전달) 흐름을 모두 보여주는 핵심적인 예시입니다.

4. 진단평가 (OX 퀴즈)

1. IoT 시스템 설계의 첫 단계는 필요한 센서와 액추에이터를 구매하는 것이다. (X)

상세 풀이: 첫 단계는 '문제 정의'입니다. 해결하고자 하는 문제가 무엇인지를 명확히 해야만, 그에 맞는 적절한 센서와 액추에이터를 선택할 수 있습니다.

2. 프로토타이핑은 실제 제품을 만들기 전에 핵심 기능을 간단히 구현하여 아이디어를 검증하는 과정이다. (O)

상세 풀이: 프로토타이핑을 통해 복잡한 전체 시스템을 만들기 전에 설계의 오류를 미리 발견하고 수정할 수 있어, 개발 비용과 시간을 크게 절약할 수 있습니다.

3. 시스템 아키텍처 설계 단계에서는 각 구성 요소의 역할과 데이터 흐름을 정의한다. (O)

6. 코딩 실습 예제 (10개)

1. 요구사항을 코드로 변환하기

"수온이 24도에서 26도 사이를 유지해야 한다"는 요구사항이 있습니다. 현재 수온을 인자로 받아 "정상", "온도 낮음 (히터 켜기)", "온도 높음 (냉각기 켜기)" 중 하나를 반환하는 함수 control_water_temp(current_temp)를 작성하세요.

def control_water_temp(current_temp):
    """'수온은 24~26도 유지' 요구사항을 코드로 구현합니다."""
    LOWER_BOUND = 24.0
    UPPER_BOUND = 26.0
    print(f"현재 수온: {current_temp}°C (요구사항: {LOWER_BOUND}~{UPPER_BOUND}°C)")
    if current_temp < LOWER_BOUND:
        return "온도 낮음 (히터 켜기)"
    elif current_temp > UPPER_BOUND:
        return "온도 높음 (냉각기 켜기)"
    else:
        return "정상"

# 테스트
print(f"판단 결과: {control_water_temp(23.5)}")
print(f"판단 결과: {control_water_temp(25.0)}")
print(f"판단 결과: {control_water_temp(26.5)}")

상세 풀이 및 설명: 이 코드는 "수온은 24도에서 26도 사이를 유지해야 한다"는 자연어 요구사항을 컴퓨터가 이해할 수 있는 논리적인 코드로 변환하는 가장 기본적인 과정을 보여줍니다. LOWER_BOUND와 UPPER_BOUND 상수를 사용하여 요구사항의 핵심 기준값들을 명시적으로 정의합니다. 이는 코드의 가독성을 높이고 향후 기준값이 변경될 때 쉽게 수정할 수 있게 합니다. if-elif-else 구문을 사용하여 현재 온도(current_temp)를 기준값과 비교하고, 세 가지 가능한 상태("낮음", "정상", "높음") 중 하나로 판단하여 그에 맞는 행동 지침을 문자열로 반환합니다. 이것이 규칙 엔진의 가장 단순한 형태입니다.

2. 간단한 디바이스 클래스 설계

고유 ID와 현재 상태("ON"/"OFF")를 가지는 SimpleSwitch 클래스를 작성하세요. turn_on(), turn_off(), get_status() 메서드를 포함하여 기본적인 액추에이터의 동작을 모델링하세요.

class SimpleSwitch:
    """ON/OFF 상태를 가지는 간단한 스위치 액추에이터 모델"""
    def __init__(self, device_id):
        self.id = device_id
        self._is_on = False # 내부 상태, 초기값은 'OFF'
        print(f"스위치({self.id})가 생성되었습니다.")
    
    def turn_on(self):
        self._is_on = True
        print(f"ACTION ({self.id}): 스위치를 ON합니다.")
        
    def turn_off(self):
        self._is_on = False
        print(f"ACTION ({self.id}): 스위치를 OFF합니다.")

    def get_status(self):
        """외부에서 현재 상태를 조회할 수 있는 메서드"""
        return "ON" if self._is_on else "OFF"

# 테스트
desk_light_switch = SimpleSwitch("desk-light")
print(f"초기 상태: {desk_light_switch.get_status()}")
desk_light_switch.turn_on()
print(f"현재 상태: {desk_light_switch.get_status()}")

상세 풀이 및 설명: SimpleSwitch 클래스는 IoT 시스템의 가장 말단에 있는 액추에이터 '사물'을 객체 지향적으로 모델링한 것입니다. __init__ 생성자는 객체가 처음 만들어질 때 호출되며, 고유 id와 초기 상태(_is_on = False)를 설정합니다. turn_on, turn_off는 외부(플랫폼, 사용자 앱 등)로부터 제어 명령을 받을 때 호출되는 '메서드'입니다. 이 메서드들은 내부 상태(_is_on)를 변경합니다. get_status는 외부에서 이 사물의 현재 상태를 안전하게 조회할 수 있도록 제공하는 '인터페이스'입니다. 내부 변수 _is_on을 직접 노출하는 것보다 이렇게 메서드를 통해 제공하는 것이 더 좋은 설계 방식입니다.

3. 상태 기계(State Machine) 시뮬레이션

'AUTO', 'MANUAL_ON', 'MANUAL_OFF' 세 가지 상태를 가지는 LightSystem 클래스를 작성하세요. set_mode(mode) 메서드로 상태를 변경할 수 있게 하세요. get_current_state() 메서드로 현재 상태를 조회할 수 있게 만드세요.

class LightSystem:
    """'AUTO', 'MANUAL_ON', 'MANUAL_OFF' 상태를 가지는 조명 시스템"""
    def __init__(self):
        self._valid_states = {'AUTO', 'MANUAL_ON', 'MANUAL_OFF'}
        self._current_state = 'AUTO' # 초기 상태
        print(f"조명 시스템 시작. 초기 모드: {self._current_state}")

    def set_mode(self, mode):
        # 유효한 모드인지 먼저 확인
        if mode in self._valid_states:
            self._current_state = mode
            print(f"모드를 '{self._current_state}'(으)로 변경합니다.")
        else:
            print(f"오류: '{mode}'는 유효하지 않은 모드입니다.")

    def get_current_state(self):
        return self._current_state

# 테스트
light_sys = LightSystem()
print(f"현재 모드: {light_sys.get_current_state()}")
light_sys.set_mode('MANUAL_ON')
print(f"현재 모드: {light_sys.get_current_state()}")
light_sys.set_mode('INVALID_MODE') # 오류 발생

상세 풀이 및 설명: 이 코드는 복잡한 시스템의 동작을 명확하게 정의하는 '상태 기계' 설계 방식을 보여줍니다. _valid_states 집합(set)은 이 시스템이 가질 수 있는 모든 상태를 미리 정의합니다. 이를 통해 존재하지 않는 상태로 시스템이 빠지는 것을 방지합니다. _current_state 변수는 시스템의 현재 상태를 저장합니다. set_mode 메서드는 외부 이벤트(예: 사용자가 앱에서 모드 변경 버튼을 누름)에 의해 상태가 다른 상태로 '전이(Transition)'되는 과정을 나타냅니다. 상태를 변경하기 전에 유효성을 검사하는 로직이 포함되어 시스템의 안정성을 높입니다.

4. 펌웨어와 플랫폼의 데이터 교환

3번의 LightSystem이 현재 상태를 1초마다 플랫폼으로 전송하는 것을 시뮬레이션하세요. Platform 클래스는 receive_status(device_id, state) 메서드를 가져, 수신된 상태를 화면에 출력하도록 작성하세요.

import time

class SimplePlatform:
    def receive_status(self, device_id, state):
        print(f"플랫폼 수신: '{device_id}'의 현재 상태는 '{state}'입니다.")

class ReportingLightSystem(LightSystem): # 위 3번 클래스 상속
    def __init__(self, device_id, platform):
        super().__init__() # 부모 클래스의 __init__ 호출
        self.id = device_id
        self.platform = platform

    def report_status_periodically(self, interval=2, count=3):
        """일정 주기로 자신의 상태를 플랫폼에 보고합니다."""
        print(f"\n--- {self.id}: 주기적인 상태 보고 시작 ---")
        for _ in range(count):
            current_state = self.get_current_state()
            print(f"디바이스({self.id}): 현재 상태 '{current_state}'를 플랫폼에 전송...")
            # 데이터 교환 발생 (메서드 호출로 시뮬레이션)
            self.platform.receive_status(self.id, current_state)
            time.sleep(interval)

# 테스트
platform = SimplePlatform()
light_sys = ReportingLightSystem("living-room-light", platform)
light_sys.report_status_periodically()

상세 풀이 및 설명: 이 코드는 디바이스(펌웨어)와 플랫폼이 어떻게 통신하는지를 보여줍니다. ReportingLightSystem은 LightSystem을 상속받아 기존 기능에 '통신' 기능을 추가했습니다. 생성자에서 통신 대상인 platform 객체를 인자로 받습니다. report_status_periodically 메서드는 펌웨어의 무한 루프 안에서 주기적으로 통신이 일어나는 것을 시뮬레이션합니다. self.platform.receive_status(...): 이 부분이 바로 데이터 전송, 즉 '상향링크(Uplink)'를 나타냅니다. 디바이스가 플랫폼의 특정 메서드(API 엔드포인트에 해당)를 호출하여 자신의 상태 데이터를 전달합니다. platform의 receive_status 메서드는 이 데이터를 받아 처리(여기서는 출력)하는 서버의 동작을 나타냅니다.

5. 복합 규칙 엔진 로직

스마트 창문 시스템을 위한 규칙 엔진 함수 window_rule_engine(data)를 작성하세요. data는 {"dust_level": 30, "is_raining": False} 형태입니다. 규칙: "미세먼지 농도가 50을 초과하고, 비가 오지 않으면 창문을 닫으세요"를 출력하세요. "비가 오면, 미세먼지 농도와 상관없이 창문을 닫으세요"를 출력하세요. 그 외는 "창문을 열어 환기하세요"를 출력하세요.

def window_rule_engine(data):
    """미세먼지와 강수 여부에 따라 창문 제어를 결정하는 규칙 엔진"""
    dust = data.get("dust_level", 0)
    raining = data.get("is_raining", False)
    print(f"\n입력: 미세먼지={dust}, 강수여부={raining}")
    
    # 규칙 1: 비가 오면 다른 조건과 상관없이 무조건 창문을 닫는다. (우선순위가 가장 높음)
    if raining:
        return "비가 오므로 창문을 닫으세요."
    
    # 규칙 2: 비가 안 올 때, 미세먼지 농도가 50을 초과하면 창문을 닫는다.
    if dust > 50:
        return "미세먼지 농도가 높으므로 창문을 닫으세요."

    # 위의 모든 규칙에 해당하지 않으면 창문을 연다.
    return "창문을 열어 환기하세요."

# 테스트
print(window_rule_engine({"dust_level": 20, "is_raining": False})) # 열기
print(window_rule_engine({"dust_level": 80, "is_raining": False})) # 닫기
print(window_rule_engine({"dust_level": 20, "is_raining": True})) # 닫기

상세 풀이 및 설명: 이 코드는 여러 센서 값을 조합하여 하나의 결정을 내리는 '복합 규칙'을 구현한 것입니다. if raining:을 가장 먼저 검사함으로써, 비가 오는 상황을 다른 모든 조건보다 우선하여 처리했습니다. 이처럼 규칙 간의 '우선순위'를 정하는 것은 규칙 엔진 설계에서 매우 중요합니다. data.get('key', default_value)를 사용하여, 만약 data 딕셔너리에 해당 키가 없더라도 프로그램이 오류 없이 기본값을 바탕으로 동작하도록 안정성을 확보했습니다. 함수의 최종 반환값은 스마트 창문(액추에이터)에 전달될 제어 명령이 됩니다.

6. 사용자 앱의 API 호출

플랫폼에 get_all_device_statuses_api() 라는 API가 있다고 가정합니다. 이 API는 [{"id": "light-1", "status": "ON"}, ...] 형태의 JSON 문자열을 반환합니다. 이 JSON을 받아 파싱한 후, 모든 장치의 ID와 상태를 보기 좋게 출력하는 display_dashboard 함수를 작성하세요.

import json

# 플랫폼 역할을 하는 클래스
class SimplePlatform:
    def get_all_device_statuses_api(self):

        statuses = [
            {"id": "light-living-room", "status": "ON"},
            {"id": "thermostat-bedroom", "status": "22.5°C"},
            {"id": "door-lock-main", "status": "LOCKED"}

# 사용자 앱 역할을 하는 함수
def display_dashboard(platform):

    # 1. API 호출 및 JSON 응답 수신
    response_json = platform.get_all_device_statuses_api()

상세 풀이 및 설명: 이 코드는 사용자 애플리케이션과 플랫폼이 'API'를 통해 어떻게 상호작용하는지를 보여줍니다. SimplePlatform의 get_all_device_statuses_api 메서드가 바로 API입니다. 이 메서드는 약속된 형식(JSON 배열)의 데이터를 반환합니다. display_dashboard 함수는 사용자 앱의 동작을 나타냅니다. platform.get_all_device_statuses_api()를 호출하는 부분이 API를 '사용'하는 부분입니다. 앱은 수신한 JSON 문자열을 json.loads로 파싱하여 리스트와 딕셔너리로 변환한 뒤, for 루프를 돌며 각 장치의 정보를 추출하여 사용자에게 보기 좋은 형태로 출력합니다.

7. 프로토타이핑: 전체 시스템 통합 시뮬레이션

'스마트 화분'을 프로토타이핑해봅시다. SmartPot 클래스(펌웨어)는 토양 습도를 측정하여 플랫폼에 보고합니다. GardeningPlatform 클래스는 습도가 20 미만이면 SmartPot의 water_plant() 메서드를 호출합니다. 이 전체 과정을 시뮬레이션하는 코드를 작성하세요.

import random

class GardeningPlatform:
    def __init__(self):
        self.pots = {} # {pot_id: pot_object}

    def register_pot(self, pot):
        self.pots[pot.id] = pot
        print(f"플랫폼: 스마트 화분({pot.id}) 등록 완료.")

    def receive_humidity(self, pot_id, humidity):
        print(f"플랫폼: '{pot_id}'의 습도({humidity}%) 데이터 수신.")
        # 규칙 엔진: 습도가 20 미만이면 물주기 명령
        if humidity < 20:
            print(f" -> 규칙 발동: '{pot_id}'가 건조합니다. 물주기를 시작합니다.")
            if pot_id in self.pots:
                self.pots[pot_id].water_plant()

class SmartPot:
    def __init__(self, pot_id, platform):
        self.id = pot_id
        self.platform = platform
        self.humidity = random.randint(10, 50)
        self.platform.register_pot(self)

    def measure_and_report(self):
        print(f"\n화분({self.id}): 현재 토양 습도({self.humidity}%)를 플랫폼에 보고합니다.")
        self.platform.receive_humidity(self.id, self.humidity)

    def water_plant(self):
        print(f" -> ACTION({self.id}): 물 공급... 습도가 25만큼 증가합니다.")
        self.humidity += 25

# 테스트 (프로토타이핑)
platform = GardeningPlatform()
pot1 = SmartPot("Rose-Pot-01", platform)

# 시뮬레이션 실행
pot1.measure_and_report() # 습도가 낮으면 물주기 실행됨
pot1.measure_and_report() # 물 준 후에는 습도가 높아져 물 안 줌

상세 풀이 및 설명: 이 코드는 '스마트 화분'이라는 하나의 완전한 IoT 시스템을 소프트웨어만으로 프로토타이핑한 것입니다. SmartPot 클래스는 펌웨어 역할을 하며, 센서 데이터(습도)를 플랫폼에 보고(measure_and_report)하고, 플랫폼의 명령을 받아 물을 주는 액추에이터 동작(water_plant)을 수행합니다. GardeningPlatform 클래스는 플랫폼 역할을 하며, 디바이스를 등록하고(register_pot), 데이터를 받아(receive_humidity), 규칙(if humidity < 20)에 따라 디바이스에 명령을 내립니다. 각 클래스의 객체를 생성하고 서로의 메서드를 호출함으로써, 실제 하드웨어가 없어도 "센싱 → 데이터 상향링크 → 플랫폼 판단 → 하향링크 명령 → 액추에이터 동작"으로 이어지는 전체 데이터 흐름과 시스템 로직을 검증할 수 있습니다.

8. 에러 처리 로직 추가

네트워크 통신이 실패하는 상황을 시뮬레이션하세요. 펌웨어가 플랫폼에 데이터를 전송할 때, 20% 확률로 "전송 실패" 메시지를 출력하고, 성공하면 "전송 성공"을 출력하는 로직을 추가하세요.

import random

def send_data_with_error_sim(platform, data):
    """20% 확률로 네트워크 전송 실패를 시뮬레이션합니다."""
    # 0.0 ~ 1.0 사이의 난수 생성
    if random.random() < 0.2:
        print(" -> 통신: 네트워크 오류! 데이터 전송에 실패했습니다.")
        return False
    else:
        print(" -> 통신: 데이터를 성공적으로 전송했습니다.")
        platform.receive(data) # 성공 시에만 플랫폼 메서드 호출
        return True

# (간단한 Platform 클래스가 있다고 가정)
class SimplePlatform:
    def receive(self, data):
        print(f" 플랫폼: 데이터 '{data}' 수신 완료.")

# 테스트
platform = SimplePlatform()
for i in range(5):
    print(f"\n[시도 {i+1}]")
    send_data_with_error_sim(platform, f"측정값-{i+1}")

상세 풀이 및 설명: IoT 시스템은 항상 안정적인 네트워크를 보장할 수 없으므로, 통신 오류 처리는 펌웨어 설계의 필수 요소입니다. random.random() < 0.2는 20%의 확률을 만들어내는 간단한 방법입니다. random.random()은 0.0에서 1.0 사이의 실수를 반환하므로, 0.2보다 작을 확률이 20%가 됩니다. if문으로 분기하여, 성공했을 경우에만 실제 플랫폼 전송 로직(platform.receive)을 호출합니다. 실제 펌웨어에서는 전송 실패 시, 데이터를 버리지 않고 잠시 저장했다가 다음 기회에 재전송(Retry)하는 로직이 추가되어야 데이터 유실을 막을 수 있습니다.

9. 원격 펌웨어 업데이트(FOTA) 시뮬레이션

Device 클래스에 firmware_version 속성을 추가하세요(초기값 1.0.0). Platform 클래스에 update_firmware(device, new_version) 메서드를 만드세요. 이 메서드는 device 객체의 firmware_version을 new_version으로 변경하고 "펌웨어 업데이트 완료" 메시지를 출력하도록 하세요.

class UpdatableDevice:
    def __init__(self, device_id, initial_version="1.0.0"):
        self.id = device_id
        self.firmware_version = initial_version

    def get_version(self):
        return self.firmware_version

    def _install_update(self, new_version):
        # 내부적으로만 호출되는 업데이트 설치 프로세스
        print(f" 장치({self.id}): 펌웨어 v{self.firmware_version} -> v{new_version} 업데이트 설치 중...")
        self.firmware_version = new_version
        print(f" 장치({self.id}): 업데이트 완료. 현재 버전: {self.firmware_version}")

class FotaPlatform:
    def trigger_firmware_update(self, device, new_version):
        print(f"\n플랫폼: 장치({device.id})에 v{new_version} 펌웨어 업데이트 명령 전송...")
        # 실제로는 네트워크를 통해 업데이트 파일을 전송하고 설치 명령을 내림
        device._install_update(new_version)

# 테스트
device = UpdatableDevice("thermostat-01")
platform = FotaPlatform()

print(f"초기 버전: {device.get_version()}")
platform.trigger_firmware_update(device, "1.1.0")
print(f"업데이트 후 버전: {device.get_version()}")

상세 풀이 및 설명: 이 코드는 IoT 플랫폼의 중요한 기능인 '원격 펌웨어 업데이트(FOTA, Firmware-Over-The-Air)'를 시뮬레이션합니다. UpdatableDevice 클래스는 자신의 firmware_version 속성을 가집니다. _install_update는 외부에서 직접 호출하는 것이 아니라, 플랫폼의 정식 명령을 통해서만 실행되어야 하는 내부 동작이므로 밑줄(_)로 시작하는 관례를 따랐습니다. FotaPlatform의 trigger_firmware_update 메서드는 플랫폼이 특정 기기(device)를 대상으로 업데이트를 실행하는 하향링크 명령을 나타냅니다. 이 기능을 통해 수백만 개의 기기를 물리적으로 방문하지 않고도, 버그를 수정하거나 새로운 기능을 추가하는 등 원격으로 유지보수가 가능해집니다.

10. 시스템 요구사항 체크리스트

requirements = {"auto_lighting": True, "remote_control": True, "power_saving_mode": False} 와 같은 딕셔너리로 요구사항 목록을 만드세요. 시스템의 각 기능을 구현하는 함수를 작성하고, 함수가 호출될 때마다 해당 요구사항이 True인지 확인하는 데코레이터(decorator)를 만들어 적용해보세요. 요구사항이 False이면 "해당 기능은 비활성화되었습니다"를 출력하고 함수를 실행하지 않도록 하세요.

# 시스템 전체의 기능 활성화 여부를 관리하는 요구사항 딕셔너리
requirements = {
    "auto_lighting": True,
    "remote_control": True,
    "power_saving_mode": False
}

def check_requirement(req_name):
    """요구사항이 활성화되었는지 확인하는 데코레이터"""
    def decorator(func):
        def wrapper(*args, **kwargs):
            if requirements.get(req_name):
                # 요구사항이 True이면 원래 함수 실행
                return func(*args, **kwargs)
            else:
                # 요구사항이 False이면 경고 메시지 출력하고 실행 안 함
                print(f"알림: '{req_name}' 기능은 현재 비활성화되었습니다.")
        return wrapper
    return decorator

# --- 데코레이터 적용 ---
@check_requirement('auto_lighting')
def run_auto_lighting_logic():
    print(" -> 자동 점등 로직 실행!")

@check_requirement('remote_control')
def execute_remote_command():
    print(" -> 원격 제어 명령 실행!")

@check_requirement('power_saving_mode')
def enter_power_saving_mode():
    print(" -> 절전 모드 실행!")

# 테스트
print("--- 기능 테스트 ---")
run_auto_lighting_logic()
execute_remote_command()
enter_power_saving_mode() # 이 기능은 실행되지 않음

상세 풀이 및 설명: 이 코드는 '데코레이터(Decorator)'라는 파이썬의 고급 기능을 사용하여, 시스템의 요구사항 변경에 유연하게 대처하는 방법을 보여줍니다. requirements 딕셔너리는 시스템의 전체 기능 스위치 역할을 합니다. check_requirement 함수는 데코레이터를 생성하는 함수입니다. 데코레이터는 기존 함수의 코드를 수정하지 않으면서, 함수 실행 전후에 추가적인 로직(여기서는 요구사항 확인)을 덧붙이는 역할을 합니다. @check_requirement(...) 구문을 함수 위에 붙이면, 해당 함수가 호출될 때마다 데코레이터의 wrapper 함수가 먼저 실행됩니다. wrapper 함수는 requirements 딕셔너리를 확인하여 기능이 활성화된 경우에만 원래 함수(func)를 실행합니다. 이러한 설계는 시스템의 버전을 관리하거나(예: "프리미엄 버전에서는 power_saving_mode 사용 가능"), 기능을 켜고 끌 때 매우 유용하고 깔끔한 코드를 작성하게 해줍니다.
