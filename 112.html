1. 학습 목표

개념적 목표

  • 네트워크에 접속하기 위해 컴퓨터에 설정해야 하는 4가지 핵심 정보(IP 주소, 서브넷 마스크, 기본 게이트웨이, DNS 서버)의 역할과 필요성을 각각 설명할 수 있다.
  • 정적(Static) IP 할당과 동적(Dynamic) IP 할당(DHCP)의 차이점을 이해하고, 각각의 장단점을 설명할 수 있다.
  • 서브넷 마스크가 IP 주소에서 네트워크 ID와 호스트 ID를 구분하는 원리를 설명할 수 있다.
  • 기본 게이트웨이가 로컬 네트워크(LAN) 외부와 통신하기 위한 '관문' 역할을 함을 이해한다.
  • DNS 서버가 도메인 이름과 IP 주소를 상호 변환하는 '인터넷 주소록' 역할을 함을 설명할 수 있다.

절차적 목표

  • 주어진 IP 주소와 서브넷 마스크를 이용해 네트워크 ID를 계산하는 코드를 작성할 수 있다.
  • 두 개의 컴퓨터가 동일한 로컬 네트워크에 있는지 판별하는 프로그램을 구현할 수 있다.
  • 딕셔너리를 사용하여 DNS 서버의 동작(도메인->IP 변환)을 간단히 시뮬레이션할 수 있다.
  • Python 클래스를 이용하여 컴퓨터의 네트워크 설정 정보(IP, 서브넷, 게이트웨이, DNS)를 하나의 객체로 모델링할 수 있다.

태도적 목표

  • 네트워크 연결 문제 발생 시, 설정 정보(IP, 서브넷, 게이트웨이, DNS)를 체계적으로 점검하고 해결하려는 논리적 문제 해결 태도를 기른다.
  • 자동화(DHCP) 기술이 복잡한 설정을 단순화하여 사용자 편의성을 높이는 원리를 이해하고, 자동화의 가치를 존중한다.
  • 눈에 보이지 않는 네트워크 구성 요소들의 상호작용을 추상적으로 이해하고, 시스템적 사고방식을 함양한다.

2. 핵심 개념 상세 설명

IP 주소 (IP Address): (복습) 네트워크에 연결된 각 장치를 식별하는 논리적 주소.

정적 IP (Static IP): 사용자가 컴퓨터에 IP 주소를 직접 수동으로 입력하고 고정하여 사용하는 방식. 주소가 바뀌지 않아 서버 컴퓨터처럼 항상 같은 주소로 접속해야 하는 경우에 유리하지만, 설정이 번거롭고 IP 주소 관리가 어렵다.

동적 IP (Dynamic IP)와 DHCP: 대부분의 일반 사용자가 사용하는 방식. 컴퓨터가 네트워크에 접속할 때마다 DHCP(Dynamic Host Configuration Protocol) 서버가 사용 가능한 IP 주소를 자동으로 할당해준다. 사용이 편리하고 IP 주소를 효율적으로 관리할 수 있다.

서브넷 마스크 (Subnet Mask): IP 주소와 쌍을 이루는 32비트 값으로, 해당 IP 주소가 속한 네트워크의 '크기'와 '범위'를 정의하는 역할을 한다. IP 주소에서 어디까지가 네트워크 ID(소속 그룹을 나타내는 부분)이고 어디부터가 호스트 ID(그룹 내 개별 장치를 나타내는 부분)인지를 구분하는 '가면(Mask)'과 같다.

예시: IP 주소가 192.168.1.100이고 서브넷 마스크가 255.255.255.0이라면, 앞의 세 부분(192.168.1)이 네트워크 ID가 되고 마지막 부분(.100)이 호스트 ID가 된다. 즉, 192.168.1.x를 사용하는 모든 컴퓨터는 같은 로컬 네트워크(LAN)에 있다고 판단한다.

기본 게이트웨이 (Default Gateway): 현재 내가 속한 로컬 네트워크(LAN)에서 다른 외부 네트워크(WAN, 인터넷)로 나가는 '출입구' 또는 '관문'. 컴퓨터가 통신하려는 목적지 IP 주소가 자신이 속한 LAN 안에 없을 때, 모든 데이터를 이 게이트웨이로 보낸다. 일반적으로 가정이나 사무실의 인터넷 공유기(라우터)의 IP 주소가 게이트웨이 주소가 된다.

DNS 서버 (Domain Name System Server): 사람이 기억하기 쉬운 도메인 이름(예: www.google.com)을 컴퓨터가 이해하는 IP 주소(예: 172.217.25.196)로 변환해주는 '인터넷 전화번호부' 역할을 하는 서버. 만약 DNS 서버 주소가 잘못 설정되면, 인터넷 주소창에 www.google.com을 입력해도 해당 사이트의 IP 주소를 찾지 못해 접속에 실패하게 된다.

(전개) 핵심 개념 탐구 (30분)

내 컴퓨터의 주소 정하기: IP 주소와 DHCP (8분)

컴퓨터에 IP 주소를 설정하는 두 가지 방법, 정적 할당과 동적 할당(DHCP)을 소개합니다. 정적 할당은 '지정 좌석', 동적 할당은 영화관에 들어갈 때마다 직원이 '빈 좌석'을 안내해주는 것에 비유합니다. 서버는 지정 좌석, 우리들의 노트북은 빈 좌석 안내가 더 효율적임을 설명합니다.

실생활 예시 코드: DHCP 서버가 IP 주소 풀(Pool)에서 사용 가능한 주소를 할당해주는 과정을 간단히 시뮬레이션합니다.

# 유전학 연구실의 DHCP 서버 시뮬레이션
ip_pool = ["192.168.10.101", "192.168.10.102", "192.168.10.103"]
assigned_ips = set()

def request_ip(mac_address):
    if not ip_pool:
        return "사용 가능한 IP가 없습니다."
    
    new_ip = ip_pool.pop(0) # 풀에서 하나를 꺼냄
    assigned_ips.add(new_ip)
    print(f"장비({mac_address})에 IP {new_ip}를 할당합니다.")
    return new_ip

print("멘델의 컴퓨터가 네트워크에 접속합니다.")
mendel_pc_ip = request_ip("0A-1B-2C-3D-4E-AA")
print("모건의 컴퓨터가 네트워크에 접속합니다.")
morgan_pc_ip = request_ip("0A-1B-2C-3D-4E-BB")

print(f"\n남은 IP 풀: {ip_pool}")
print(f"할당된 IP 목록: {assigned_ips}")

상세 설명: 이 코드는 DHCP의 핵심 원리인 'IP 주소 대여'를 보여줍니다. ip_pool은 DHCP 서버가 나눠줄 수 있는 IP 주소 목록입니다. request_ip 함수가 호출될 때마다(새로운 컴퓨터가 접속할 때마다) 목록에서 IP를 하나 꺼내(pop)주고, 할당된 IP는 assigned_ips에 기록하여 중복 할당을 막습니다. 이를 통해 학생들은 동적 IP 할당의 자동화된 프로세스를 직관적으로 이해할 수 있습니다.

우리 동네는 어디까지? 서브넷 마스크 (10분)

IP 주소만으로는 어디까지가 '우리 동네(LAN)'이고 어디부터가 '다른 동네(WAN)'인지 알 수 없음을 지적합니다. 서브넷 마스크 255.255.255.0을 예로 들어, 255에 해당하는 부분이 네트워크 ID, 0에 해당하는 부분이 호스트 ID임을 설명합니다. 컴퓨터가 IP와 서브넷 마스크에 비트 AND 연산을 하여 네트워크 ID를 계산하는 원리를 설명합니다.

수학/과학 예시 코드: IP 주소와 서브넷 마스크를 입력받아, 비트 AND 연산을 통해 네트워크 ID를 계산하는 함수를 작성합니다.

def get_network_id(ip, subnet_mask):
    ip_parts = [int(p) for p in ip.split('.')]
    mask_parts = [int(p) for p in subnet_mask.split('.')]
    
    net_id_parts = []
    for i in range(4):
        # 비트 AND 연산 (&)
        net_id_parts.append(str(ip_parts[i] & mask_parts[i]))
        
    return ".".join(net_id_parts)

my_ip = "192.168.1.100"
my_mask = "255.255.255.0"
network_address = get_network_id(my_ip, my_mask)

print(f"IP 주소: {my_ip}")
print(f"서브넷 마스크: {my_mask}")
print(f"계산된 네트워크 ID: {network_address}")

상세 설명: 이 코드는 서브넷 마스크의 실제 역할을 명확히 보여줍니다. IP 주소와 서브넷 마스크의 각 부분을 숫자로 변환한 뒤, 파이썬의 비트 AND 연산자 &를 사용합니다. 192 & 255는 192, 100 & 0은 0이 됩니다. 이 연산을 통해 IP 주소에서 호스트 ID 부분이 '가려져(Masking)' 네트워크 ID만 남게 되는 과정을 학생들이 직접 눈으로 확인할 수 있습니다. 이는 컴퓨터 내부에서 일어나는 논리 연산을 가시화한 것입니다.

외부로 나가는 문: 기본 게이트웨이 (6분)

"같은 동네라면? 직접 찾아간다. 다른 동네라면? 모든 짐(패킷)을 '기본 게이트웨이'라는 대문 앞에 가져다 놓는다." 라고 비유합니다.

실생활 예시 코드: 목적지 IP가 로컬 네트워크에 있는지, 아니면 게이트웨이로 보내야 하는지 판단하는 함수를 작성합니다.

# 위 get_network_id 함수가 필요합니다.
def should_use_gateway(my_ip, dest_ip, subnet_mask):
    my_net_id = get_network_id(my_ip, subnet_mask)
    dest_net_id = get_network_id(dest_ip, subnet_mask)
    
    if my_net_id == dest_net_id:
        print(f"목적지({dest_ip})는 같은 네트워크에 있습니다. 직접 통신합니다.")
        return False
    else:
        print(f"목적지({dest_ip})는 외부 네트워크에 있습니다. 게이트웨이를 사용합니다.")
        return True

my_ip = "192.168.1.100"
subnet_mask = "255.255.255.0"
friend_pc_ip = "192.168.1.50" # 같은 LAN
google_server_ip = "8.8.8.8" # 다른 LAN (WAN)

should_use_gateway(my_ip, friend_pc_ip, subnet_mask)
should_use_gateway(my_ip, google_server_ip, subnet_mask)

상세 설명: 이 코드는 게이트웨이의 역할이 '조건부'임을 보여줍니다. should_use_gateway 함수는 내 IP와 목적지 IP의 네트워크 ID를 각각 계산합니다. 두 네트워크 ID가 같으면 로컬 통신으로 판단하고, 다르면 외부 통신으로 판단하여 게이트웨이를 사용해야 한다고 알려줍니다. 이는 컴퓨터가 네트워크 통신을 할 때 내리는 가장 기본적인 의사결정 과정을 코드로 구현한 것입니다.

인터넷 전화번호부: DNS 서버 (6분)

www.google.com 같은 도메인 이름의 편리함을 상기시키고, 컴퓨터는 결국 IP 주소로 통신함을 강조합니다. DNS 서버가 이 둘 사이를 변환해주는 '통역사' 또는 '주소록' 역할을 한다고 설명합니다.

실생활 예시 코드: 딕셔너리를 사용하여 DNS 서버가 도메인 이름으로 IP 주소를 찾아주는 과정을 시뮬레이션합니다.

# NASA의 일부 도메인에 대한 가상 DNS 서버
dns_server_db = {
    "www.nasa.gov": "23.215.137.147",
    "api.nasa.gov": "52.4.220.199",
    "images.nasa.gov": "104.85.118.157"
}

def dns_lookup(domain_name):
    # .get() 메서드로 도메인에 해당하는 IP를 찾음. 없으면 None 반환
    ip_address = dns_server_db.get(domain_name)
    if ip_address:
        print(f"DNS 질의: {domain_name} -> IP: {ip_address}")
        return ip_address
    else:
        print(f"DNS 질의 실패: {domain_name}을(를) 찾을 수 없습니다.")
        return None

dns_lookup("www.nasa.gov")
dns_lookup("www.spacex.com") # DB에 없는 도메인

상세 설명: 이 코드는 DNS의 핵심 기능인 '이름 해석(Name Resolution)'을 딕셔너리의 키-값 검색으로 단순화하여 보여줍니다. dns_server_db 딕셔너리가 DNS 서버의 데이터베이스 역할을 합니다. dns_lookup 함수는 사용자가 요청한 도메인 이름을 키로 사용하여 딕셔너리에서 IP 주소(값)를 찾아 반환합니다. 이를 통해 학생들은 복잡한 DNS 시스템의 기본 원리가 간단한 '찾기' 동작에 기반함을 이해할 수 있습니다.

(마무리) 수업을 마치며 (5분)

오늘 배운 4가지 핵심 설정(IP/DHCP, 서브넷 마스크, 게이트웨이, DNS)이 어떻게 상호작용하며 인터넷 연결을 가능하게 하는지 종합적으로 요약합니다.

"우리가 세상을 이해하는 유일한 방법은 그것을 재창조하는 것이다. 원리를 이해할 때까지는 조작할 수 없고, 조작할 수 있을 때까지는 이해했다고 할 수 없다." - 괴테 (의역 및 조합)

"여러분, 오늘 우리는 네트워크 설정을 직접 '조작'하는 원리를 배웠습니다. IP 주소를 설정하고, 서브넷 마스크로 우리 동네를 정의하고, 게이트웨이로 나가는 문을 지정하고, DNS로 길을 물어보는 이 모든 과정이 바로 네트워크를 '재창조'하는 행위입니다. 오늘 배운 이 원리들을 통해, 여러분은 이제 보이지 않는 네트워크의 세계를 더 깊이 이해하고 다룰 수 있는 힘을 얻게 된 것입니다. 수고 많으셨습니다."

4. 진단평가 (OX 퀴즈)

1. DHCP는 사용자가 IP 주소를 직접 수동으로 설정하는 방식이다. (X)

상세 풀이: DHCP(Dynamic Host Configuration Protocol)는 컴퓨터가 네트워크에 접속할 때 IP 주소, 서브넷 마스크 등의 정보를 자동으로 할당받는 방식입니다. 수동 설정 방식은 '정적(Static) IP' 방식입니다.

2. 서브넷 마스크는 IP 주소에서 네트워크 부분과 호스트 부분을 구분하는 역할을 한다. (O)

상세 풀이: 서브넷 마스크는 IP 주소에 비트 AND 연산을 적용하여, 어디까지가 모든 장치가 공유하는 네트워크 주소(네트워크 ID)이고 어디부터가 각 장치의 고유 주소(호스트 ID)인지를 구분해줍니다.

3. 다른 웹사이트에 접속하기 위해 사용하는 인터넷 공유기의 IP 주소를 보통 DNS 서버 주소라고 한다. (X)

상세 풀이: 외부 네트워크로 나가는 관문 역할을 하는 인터넷 공유기의 주소는 '기본 게이트웨이(Default Gateway)' 주소입니다. DNS 서버는 도메인 이름을 IP 주소로 변환해주는 역할을 하는 별도의 서버 주소입니다.

4. 컴퓨터에 DNS 서버 주소가 설정되어 있지 않으면, IP 주소(예: 172.217.25.196)를 직접 입력해서는 웹사이트에 접속할 수 없다. (X)

상세 풀이: DNS 서버는 도메인 이름(www.google.com)을 IP 주소로 '변환'해주는 역할만 합니다. 만약 IP 주소를 이미 알고 있다면, DNS 서버를 거칠 필요 없이 IP 주소를 직접 주소창에 입력하여 사이트에 접속할 수 있습니다.

5. 정적 IP는 주소가 바뀌지 않아 서버 운영에 적합하다. (O)

상세 풀이: 서버는 항상 같은 주소로 외부에서 접속할 수 있어야 하므로, 주소가 계속 바뀌는 동적 IP보다는 고정된 정적 IP를 사용하는 것이 안정적이고 관리에 용이합니다.

6. 192.168.0.10과 192.168.1.10은 서브넷 마스크가 255.255.255.0일 때 같은 로컬 네트워크에 있다. (X)

상세 풀이: 서브넷 마스크가 255.255.255.0이면 IP 주소의 앞 세 부분(192.168.x)이 네트워크 ID가 됩니다. 두 IP는 세 번째 부분(0과 1)이 다르므로 서로 다른 네트워크에 속합니다.

7. 기본 게이트웨이는 우리 집 컴퓨터가 옆방 컴퓨터와 통신할 때 반드시 필요하다. (X)

상세 풀이: 기본 게이트웨이는 '외부' 네트워크와 통신할 때 사용되는 관문입니다. 옆방 컴퓨터처럼 같은 로컬 네트워크(LAN) 내에 있는 장치와 통신할 때는 게이트웨이를 거치지 않고 직접 통신합니다.

8. DHCP 서버는 IP 주소뿐만 아니라 서브넷 마스크, 게이트웨이, DNS 서버 정보도 함께 제공해줄 수 있다. (O)

상세 풀이: DHCP는 단순히 IP 주소만 할당하는 것이 아니라, 해당 네트워크를 사용하는 데 필요한 모든 정보(IP 주소, 서브넷 마스크, 기본 게이트웨이 주소, DNS 서버 주소 등)를 패키지로 묶어 클라이언트에게 전달해줍니다.

9. 서브넷 마스크 255.255.0.0은 255.255.255.0보다 더 많은 수의 호스트(컴퓨터)를 포함할 수 있는 네트워크를 정의한다. (O)

상세 풀이: 255.255.0.0은 호스트 ID로 사용할 수 있는 부분이 뒤의 두 부분(16비트)인 반면, 255.255.255.0은 마지막 한 부분(8비트)만 호스트 ID로 사용할 수 있습니다. 따라서 255.255.0.0이 훨씬 더 큰 네트워크(더 많은 호스트)를 의미합니다.

10. 도메인 이름(예: www.naver.com)을 사용하는 주된 이유는 IP 주소보다 보안에 더 뛰어나기 때문이다. (X)

상세 풀이: 도메인 이름을 사용하는 주된 이유는 숫자로 된 IP 주소보다 사람이 기억하고 사용하기 훨씬 편리하기 때문입니다. 보안과는 직접적인 관련이 적습니다.

5. 형성평가 (5지선다형)

1. PC의 네트워크 설정을 확인했더니 IP 주소가 169.254.x.x로 시작했다. 이 현상의 가장 가능성 높은 원인은?

  1. 정상적으로 인터넷에 연결된 상태이다.
  2. DNS 서버 설정이 잘못되었다.
  3. 정적 IP가 성공적으로 설정되었다.
  4. DHCP 서버로부터 IP 주소를 할당받지 못했다.
  5. 게이트웨이 주소가 존재하지 않는다.

정답: ④

상세 풀이: 169.254.x.x 대역의 주소는 APIPA(Automatic Private IP Addressing) 주소라고 합니다. 이는 윈도우 운영체제가 DHCP 서버를 찾았지만 응답을 받지 못했을 때, 임시로 스스로에게 할당하는 링크-로컬 주소입니다. 즉, DHCP 서버와 통신에 실패했음을 의미합니다.

2. 다음 중 외부 인터넷(WAN)과 통신하기 위해 반드시 정확하게 설정되어야 하는 정보는?

  1. 내 컴퓨터 이름
  2. 서브넷 마스크
  3. 기본 게이트웨이
  4. MAC 주소
  5. 블루투스 PIN 번호

정답: ③

상세 풀이: 기본 게이트웨이는 로컬 네트워크(LAN)에서 외부 네트워크(WAN)로 나가는 유일한 '출구'입니다. 이 주소가 잘못 설정되면 컴퓨터는 인터넷 세상으로 나가는 길을 찾지 못해 외부와 통신할 수 없습니다.

3. 인터넷 브라우저 주소창에 www.google.com을 입력하면 접속되지만, 216.58.220.142(구글의 IP 중 하나)를 입력하면 접속되지 않는다. 어떤 설정에 문제가 있을 가능성이 가장 높은가?

  1. IP 주소 할당 문제
  2. 서브넷 마스크 설정 오류
  3. 기본 게이트웨이 설정 오류
  4. DNS 서버 설정 오류
  5. 위 보기 중에는 답이 없음

정답: ⑤

상세 풀이: 반대의 경우, 즉 도메인 이름으로 접속이 안 되고 IP 주소로만 접속이 된다면 DNS 서버의 문제입니다. 하지만 이 경우는 IP 주소로도 직접 접속이 안 되므로, 이름 변환(DNS) 단계의 문제가 아니라 그 이후의 통신 과정, 즉 IP 주소를 이용한 실제 연결 과정에 문제가 있는 것입니다. 게이트웨이 설정 오류(③)나 방화벽 문제 등일 가능성이 높습니다. 보기 중에는 정답이 없습니다. (문제 의도를 파악하는 함정 문제) (만약 문제가 '도메인은 안되고 IP는 될 때' 였다면 정답은 ④번입니다.)

4. IP 주소가 192.168.30.5이고 서브넷 마스크가 255.255.255.0일 때, 이 컴퓨터의 네트워크 ID는 무엇인가?

  1. 192.168.30.5
  2. 255.255.255.0
  3. 192.168.30.0
  4. 0.0.0.5
  5. 192.168.0.0

정답: ③

상세 풀이: 서브넷 마스크가 255.255.255.0이라는 것은 IP 주소의 앞 3부분까지가 네트워크 ID이고 마지막 부분은 호스트 ID라는 의미입니다. 따라서 네트워크 ID는 192.168.30 부분이고, 호스트 ID 부분을 0으로 채운 192.168.30.0이 네트워크 주소가 됩니다.

5. 다음 중 동적 IP 할당 방식(DHCP)의 가장 큰 장점은?

  1. 모든 컴퓨터가 고정된 IP를 사용하여 관리가 쉽다.
  2. IP 주소를 수동으로 설정하여 충돌 위험이 없다.
  3. 네트워크 관리자가 IP 주소를 일일이 할당할 필요가 없어 편리하다.
  4. 서버 컴퓨터 운영에 가장 적합한 방식이다.
  5. 한번 할당받은 IP는 절대 변하지 않는다.

정답: ③

상세 풀이: DHCP의 가장 큰 장점은 '자동화'입니다. 수많은 컴퓨터에 관리자가 직접 IP를 설정할 필요 없이, 컴퓨터가 네트워크에 접속하기만 하면 필요한 모든 정보가 자동으로 할당됩니다. 이는 관리의 편의성을 극대화하고 IP 자원을 효율적으로 사용하게 합니다.

6. "인터넷은 되는데, 특정 게임 접속만 안된다"는 문제가 발생했다. 이 때, 점검해야 할 설정과 가장 거리가 먼 것은?

  1. 해당 게임 서버의 주소
  2. 방화벽의 포트 차단 설정
  3. 공유기의 특정 서비스 차단 설정
  4. 내 컴퓨터의 서브넷 마스크 설정
  5. 게임 서버의 점검 공지

정답: ④

상세 풀이: 인터넷 자체가 잘 된다는 것은 IP 주소, 서브넷 마스크, 기본 게이트웨이, DNS 서버 등 기본적인 네트워크 설정은 올바르게 되어있다는 뜻입니다. 특정 서비스(게임)만 접속이 안 될 경우, 해당 서비스가 사용하는 특정 포트를 방화벽이나 공유기가 막고 있을 가능성(②, ③) 또는 게임 서버 자체의 문제(①, ⑤)일 가능성이 높습니다. 서브넷 마스크는 기본적인 인터넷 연결 자체에 영향을 주므로 이미 확인된 부분입니다.

7. 컴퓨터의 네트워크 설정 4요소와 그 역할을 비유한 것으로 가장 적절하지 않은 것은?

  1. IP 주소 - 집 주소
  2. 서브넷 마스크 - 우리 동네의 범위 지도
  3. 기본 게이트웨이 - 동네 밖으로 나가는 큰 길의 입구
  4. DNS 서버 - 내 컴퓨터의 CPU
  5. DHCP - 주소를 자동으로 발급해주는 동사무소 직원

정답: ④

상세 풀이: DNS 서버는 도메인 이름을 IP 주소로 바꿔주는 '인터넷 전화번호부'나 '통역사'에 비유할 수 있습니다. CPU는 컴퓨터의 연산을 담당하는 중앙 처리 장치로, 네트워크 설정과는 역할이 다릅니다. 나머지 비유는 모두 각 요소의 역할을 잘 설명하고 있습니다.

8. IP 주소 A = "172.16.10.20" 와 B = "172.16.20.30"가 있다. 서브넷 마스크가 255.255.0.0일 때, 두 컴퓨터의 관계에 대한 설명으로 옳은 것은?

  1. A와 B는 서로 다른 네트워크에 있다.
  2. A와 B는 같은 로컬 네트워크(LAN)에 있다.
  3. A가 B에게 통신하려면 반드시 게이트웨이를 거쳐야 한다.
  4. 서브넷 마스크가 잘못 설정되었다.
  5. A와 B는 같은 컴퓨터이다.

정답: ②

상세 풀이: 서브넷 마스크가 255.255.0.0이면 IP 주소의 앞 두 부분(172.16)이 네트워크 ID가 됩니다. A와 B 모두 네트워크 ID가 172.16으로 동일하므로, 같은 로컬 네트워크에 속해 있습니다. 따라서 서로 통신할 때 게이트웨이를 거치지 않고 직접 통신할 수 있습니다.

9. 다음 중 네트워크 설정 정보를 자동으로 받아오는 프로토콜은 무엇인가?

  1. TCP
  2. IP
  3. DNS
  4. HTTP
  5. DHCP

정답: ⑤

상세 풀이: DHCP(Dynamic Host Configuration Protocol)는 동적 호스트 구성 프로토콜로, 네트워크에 접속하는 클라이언트에게 IP 주소, 서브넷 마스크 등의 설정 정보를 자동으로 할당해주는 역할을 하는 프로토콜입니다.

10. 카페의 공용 와이파이에 접속할 때 내 노트북이 IP 주소를 할당받는 과정은 다음 중 어느 것에 가장 가까운가?

  1. 정적 IP 설정
  2. DHCP를 통한 동적 IP 할당
  3. DNS 서버에 직접 주소 요청
  4. MAC 주소를 IP 주소로 사용
  5. APIPA를 통한 주소 자동 생성

정답: ②

상세 풀이: 카페, 학교, 공항 등 불특정 다수가 접속하는 공용 네트워크 환경에서는 각 사용자에게 일일이 IP를 수동으로 설정해줄 수 없습니다. 따라서 카페의 공유기(라우터)가 DHCP 서버 역할을 하여 접속하는 장치들에게 IP 주소를 자동으로 분배해주는 동적 IP 할당 방식을 사용합니다.

6. 코딩 실습 예제

1. 서브넷 마스크 유효성 검사기

def validate_subnet_mask(mask_string):
    """입력된 문자열이 유효한 서브넷 마스크인지 판별합니다."""
    parts = mask_string.split('.')
    if len(parts) != 4: return False
    
    full_binary_str = ""
    for part in parts:
        if not part.isdigit(): return False
        num = int(part)
        if not 0 <= num <= 255: return False
        full_binary_str += bin(num)[2:].zfill(8)

    # 이진수 표현에서 '0'이 나온 이후에 '1'이 나오면 안됨
    if '01' in full_binary_str:
        return False
    return True

# 테스트
mask1 = "255.255.255.0"
mask2 = "255.255.240.0"
mask3 = "255.255.0.255"
print(f"'{mask1}'는 유효한 서브넷 마스크인가? {validate_subnet_mask(mask1)}")
print(f"'{mask2}'는 유효한 서브넷 마스크인가? {validate_subnet_mask(mask2)}")
print(f"'{mask3}'는 유효한 서브넷 마스크인가? {validate_subnet_mask(mask3)}")

상세 풀이 및 설명: 먼저 IP 주소처럼 4개의 부분으로 나뉘고, 각 부분이 0-255 사이의 숫자인지 기본적인 형식을 검사합니다. full_binary_str = "" 빈 문자열을 만들고, 각 숫자 부분을 8자리 이진수로 변환하여 모두 이어 붙여 32자리 이진수 문자열을 만듭니다. 유효한 서브넷 마스크는 이진수로 표현했을 때 1이 연속으로 나타나다가 0이 나오기 시작하면 그 뒤는 모두 0이어야 합니다. 따라서 if '01' in full_binary_str: 조건문으로 full_binary_str 안에 '01'이라는 부분 문자열이 있는지 검사합니다. 이 패턴이 존재한다는 것은 0 다음에 1이 다시 나타났다는 뜻이므로, 유효하지 않은 서브넷 마스크입니다. 이 경우 False를 반환합니다. 모든 검사를 통과하면 True를 반환합니다.

3. 간단한 DHCP 서버 시뮬레이터

class DHCPServer:
    def __init__(self, ip_pool_start, pool_size, subnet, gateway, dns):
        start_prefix = ip_pool_start.rsplit('.', 1)[0]
        start_num = int(ip_pool_start.split('.')[-1])
        self.ip_pool = [f"{start_prefix}.{start_num + i}" for i in range(pool_size)]
        self.subnet = subnet
        self.gateway = gateway
        self.dns = dns
        self.leases = {} # {mac: ip} 형태로 임대 정보를 저장
        print("DHCP 서버가 시작되었습니다.")

    def request_network_config(self, mac_address):
        if mac_address in self.leases:
            ip = self.leases[mac_address]
            print(f"갱신 요청: {mac_address} -> {ip}")
            return {'ip': ip, 'subnet': self.subnet, 'gateway': self.gateway, 'dns': self.dns}

        if not self.ip_pool:
            return "오류: 사용 가능한 IP가 없습니다."
        
        new_ip = self.ip_pool.pop(0)
        self.leases[mac_address] = new_ip
        print(f"신규 할당: {mac_address} -> {new_ip}")
        return {'ip': new_ip, 'subnet': self.subnet, 'gateway': self.gateway, 'dns': self.dns}

# 테스트

pc1_config = server.request_network_config("AA-AA-AA-AA-AA-AA")
pc2_config = server.request_network_config("BB-BB-BB-BB-BB-BB")
pc1_config_renew = server.request_network_config("AA-AA-AA-AA-AA-AA") # 갱신 요청
print("\nPC1 설정:", pc1_config)

상세 풀이 및 설명: DHCPServer 클래스는 IP 주소 풀, 그리고 나머지 네트워크 설정 정보를 생성자에서 미리 받아 저장합니다. leases 딕셔너리는 어떤 MAC 주소에 어떤 IP를 할당했는지 기록하는 '임대 장부'입니다. request_network_config 함수는 MAC 주소를 인자로 받습니다. 먼저 leases 딕셔너리에서 해당 MAC 주소가 이미 IP를 할당받았는지 확인합니다. 만약 그렇다면, 기존 IP 정보를 담아 '갱신' 응답을 보냅니다. 신규 요청일 경우, ip_pool에 남은 IP가 있는지 확인합니다. self.ip_pool.pop(0)으로 리스트의 첫 번째 IP를 꺼내 new_ip에 저장합니다. self.leases[mac_address] = new_ip 코드로 임대 장부에 MAC 주소와 할당된 IP를 기록합니다. 할당된 new_ip와 서버가 가지고 있던 나머지 설정 정보를 딕셔너리로 묶어 반환합니다.

4. IP 주소와 서브넷 마스크로 방송 주소(Broadcast Address) 계산하기

def get_broadcast_address(ip, subnet_mask):
    """IP와 서브넷 마스크를 이용해 방송 주소를 계산합니다."""
    ip_parts = [int(p) for p in ip.split('.')]
    mask_parts = [int(p) for p in subnet_mask.split('.')]
    
    broadcast_parts = []
    for i in range(4):
        # 호스트 마스크는 서브넷 마스크의 비트를 뒤집은 것
        host_mask_part = (~mask_parts[i]) & 255
        # 브로드캐스트 주소 = IP 주소 OR 호스트 마스크
        broadcast_part = ip_parts[i] | host_mask_part
        broadcast_parts.append(str(broadcast_part))
        
    return ".".join(broadcast_parts)

# 테스트
ip = "172.16.10.20"
mask = "255.255.0.0"
bcast_addr = get_broadcast_address(ip, mask)
print(f"IP: {ip}, Mask: {mask}")
print(f"Broadcast Address: {bcast_addr}") # 예상 결과: 172.16.255.255

상세 풀이 및 설명: 방송 주소는 네트워크 ID는 그대로 두고, 호스트 ID 부분을 모두 1로 만든 주소입니다. host_mask_part = (~mask_parts[i]) & 255: 이 부분이 호스트 ID 부분을 1로 만드는 마스크를 생성하는 과정입니다. 서브넷 마스크의 비트를 반전(~)시키면 호스트 ID 부분만 1이 됩니다. (예: 255.255.255.0 -> 0.0.0.255) & 255는 파이썬의 비트 연산 특성상 음수가 되는 것을 방지하고 8비트 범위 내에서만 연산하기 위함입니다. broadcast_part = ip_parts[i] | host_mask_part: IP 주소와 호스트 마스크에 비트 OR 연산(|)을 수행합니다. 이 연산은 둘 중 하나라도 비트가 1이면 결과 비트가 1이 됩니다. 네트워크 ID 부분: 호스트 마스크가 0이므로 원래 IP 주소 값이 그대로 유지됩니다. (172 | 0 = 172) 호스트 ID 부분: 호스트 마스크가 255(모든 비트가 1)이므로, 원래 IP 주소 값과 상관없이 결과는 255가 됩니다. (10 | 255 = 255) 결과적으로 네트워크 ID는 유지되고 호스트 ID 부분만 모두 1로 채워진 방송 주소가 계산됩니다.

5. CIDR 표기법 변환기 (/24)

def subnet_to_cidr(subnet_mask):
    """서브넷 마스크 문자열을 CIDR 숫자 (e.g., /24)로 변환합니다."""
    parts = subnet_mask.split('.')
    cidr = 0
    for part in parts:
        num = int(part)
        # bin()으로 이진수로 변환하고, '1'의 개수를 센다
        cidr += bin(num).count('1')
    return cidr

# 테스트
mask1 = "255.255.255.0"
mask2 = "255.255.252.0" # /22
cidr1 = subnet_to_cidr(mask1)
cidr2 = subnet_to_cidr(mask2)
print(f"{mask1} -> /{cidr1}")
print(f"{mask2} -> /{cidr2}")

상세 풀이 및 설명: CIDR 숫자는 서브넷 마스크를 32자리 이진수로 표현했을 때 1의 총 개수입니다. 서브넷 마스크 문자열을 점으로 분리합니다. cidr = 0으로 카운터 변수를 초기화합니다. for 루프로 각 부분을 순회하며, 먼저 int()로 숫자로 변환합니다. bin(num)은 숫자를 '0b...' 형태의 이진수 문자열로 변환합니다. .count('1') 문자열 메서드는 해당 문자열 안에 '1'이 몇 개나 있는지 세어줍니다. cidr += ...: 각 부분에서 센 1의 개수를 cidr 변수에 계속 더해 나갑니다. 반복이 끝나면 cidr에는 32비트 전체에서 1의 총 개수가 담기게 되고, 이를 반환합니다.

6. 로컬 DNS 캐시 시뮬레이터

# 실제 인터넷상의 DNS 서버라고 가정
dns_server_db = { "www.google.com": "216.58.220.142", "www.github.com": "140.82.121.4" }
# 내 컴퓨터의 DNS 캐시라고 가정
dns_cache = {}

def dns_lookup_with_cache(domain):
    # 1. 로컬 캐시에 있는지 먼저 확인
    if domain in dns_cache:
        print(f"CACHE HIT: {domain} -> {dns_cache[domain]}")
        return dns_cache[domain]
    
    # 2. 캐시에 없으면, 실제 DNS 서버에 질의
    print(f"CACHE MISS: {domain}... 실제 서버에 질의합니다.")
    ip = dns_server_db.get(domain)
    
    if ip:
        # 3. 찾은 결과를 캐시에 저장
        print(f"응답 받음. 결과를 캐시에 저장합니다: {domain} -> {ip}")
        dns_cache[domain] = ip
        return ip
    else:
        return "오류: 해당 도메인을 찾을 수 없습니다."

# 테스트
print("첫 번째 google.com 조회:")
dns_lookup_with_cache("www.google.com")
print("\n두 번째 google.com 조회:")
dns_lookup_with_cache("www.google.com")

상세 풀이 및 설명: DNS 서버의 데이터베이스 역할을 하는 dns_server_db와 로컬 캐시 역할을 하는 dns_cache 딕셔너리를 각각 생성합니다. dns_lookup_with_cache 함수가 호출되면, if domain in dns_cache: 조건문으로 가장 먼저 로컬 캐시를 확인합니다. 캐시에 해당 도메인이 키로 존재하면, 서버에 묻지 않고 즉시 캐시의 값을 반환합니다. 이를 '캐시 히트(Cache Hit)'라고 합니다. 캐시에 도메인이 없으면('캐시 미스(Cache Miss)'), 실제 DNS 서버(dns_server_db)에 질의하여 IP 주소를 가져옵니다. 서버에서 성공적으로 IP를 받으면, 다음 조회를 위해 dns_cache[domain] = ip 코드로 결과를 로컬 캐시에 저장한 후 값을 반환합니다. 이 로직 덕분에 동일한 도메인을 두 번째 조회할 때부터는 훨씬 빠른 속도로 응답을 받을 수 있습니다.

7. 네트워크 설정 정보 파일로 저장하고 읽기

class NetworkConfig:
    def __init__(self, ip, subnet, gateway, dns):
        self.ip, self.subnet, self.gateway, self.dns = ip, subnet, gateway, dns
    def display_config(self):
        print(f"IP: {self.ip}, DNS: {self.dns}")

    def save_to_file(self, filename):
        with open(filename, 'w') as f:
            f.write(f"IP={self.ip}\n")
            f.write(f"SUBNET={self.subnet}\n")
            f.write(f"GATEWAY={self.gateway}\n")
            f.write(f"DNS={self.dns}\n")
        print(f"설정 정보를 '{filename}' 파일에 저장했습니다.")

    @classmethod
    def load_from_file(cls, filename):
        config_data = {}
        try:
            with open(filename, 'r') as f:
                for line in f:
                    key, value = line.strip().split('=', 1)
                    config_data[key] = value
            print(f"'{filename}' 파일에서 설정 정보를 읽어왔습니다.")
            return cls(config_data['IP'], config_data['SUBNET'], config_data['GATEWAY'], config_data['DNS'])
        except FileNotFoundError: return "오류: 파일을 찾을 수 없습니다."
        except (KeyError, ValueError): return "오류: 파일 형식이 잘못되었습니다."
# 테스트
pc1 = NetworkConfig("10.0.1.5", "255.255.255.0", "10.0.1.1", "1.1.1.1")
pc1.save_to_file("pc1_config.txt")
pc2 = NetworkConfig.load_from_file("pc1_config.txt")
if isinstance(pc2, NetworkConfig):
    print("\n파일로부터 로드된 객체 정보:")
    pc2.display_config()

상세 풀이 및 설명: save_to_file 메서드: with open(...) 구문을 사용하여 파일을 쓰기 모드('w')로 엽니다. f-string을 이용해 키=값 형식의 문자열을 만들어 f.write()로 파일에 한 줄씩 씁니다. load_from_file 메서드: @classmethod 데코레이터는 이 메서드가 특정 인스턴스가 아닌 클래스 자체에 속해 있음을 의미합니다. 따라서 pc1.load_from_file()이 아닌 NetworkConfig.load_from_file()처럼 호출할 수 있습니다. 파일을 읽기 모드('r')로 열고, 각 줄을 순회합니다. line.strip().split('=', 1): 줄 끝의 공백/줄바꿈을 제거하고, 첫 번째 나오는 =를 기준으로 문자열을 한 번만 분리하여 [키, 값] 형태의 리스트를 만듭니다. 이 키와 값을 config_data 딕셔너리에 저장합니다. 파일 읽기가 끝나면, cls(...)를 호출하여 클래스(NetworkConfig)의 새 인스턴스를 생성하고 반환합니다. cls는 @classmethod 안에서 클래스 자신을 가리킵니다. try-except 구문으로 파일이 없거나 내용이 잘못된 경우의 오류를 처리합니다.

8. IP 주소 정수 변환

def ip_to_int(ip_string):
    """IP 주소 문자열을 32비트 정수로 변환합니다."""
    parts = [int(p) for p in ip_string.split('.')]
    # 비트 시프트 연산으로 더 효율적으로 계산 가능
    integer_ip = (parts[0] << 24) + (parts[1] << 16) + (parts[2] << 8) + parts[3]
    return integer_ip

def int_to_ip(ip_int):
    """32비트 정수를 IP 주소 문자열로 변환합니다."""
    part1 = (ip_int >> 24) & 255
    part2 = (ip_int >> 16) & 255
    part3 = (ip_int >> 8) & 255
    part4 = ip_int & 255
    return f"{part1}.{part2}.{part3}.{part4}"

# 테스트
ip = "192.168.0.1"
ip_as_integer = ip_to_int(ip)
print(f"{ip} -> {ip_as_integer}")
restored_ip = int_to_ip(ip_as_integer)
print(f"{ip_as_integer} -> {restored_ip}")

상세 풀이 및 설명: ip_to_int: <<는 왼쪽 비트 시프트(shift) 연산자입니다. x << n은 x의 비트를 왼쪽으로 n번 이동시키며, 이는 x * (2^n)과 같습니다. parts[0] << 24 : 첫 번째 부분을 왼쪽으로 24비트(3바이트) 이동시킵니다. parts[1] << 16 : 두 번째 부분을 왼쪽으로 16비트(2바이트) 이동시킵니다. 이들을 모두 더하면, 각 부분이 32비트 정수의 각 바이트 위치에 정확히 자리 잡게 됩니다. int_to_ip: >>는 오른쪽 비트 시프트 연산자입니다. (ip_int >> 24) & 255 : 정수를 오른쪽으로 24비트 이동시켜 첫 바이트 부분만 남긴 후, & 255 연산으로 정확히 8비트 값만 추출합니다. 이 과정을 16비트, 8비트, 0비트에 대해 반복하여 각 부분의 숫자를 얻고, f-string으로 합쳐 IP 주소 문자열을 만듭니다.

9. 포트 스캐너 시뮬레이터

def port_scanner_simulator(host, start_port, end_port):
    """가상의 호스트에 대해 지정된 범위의 포트 상태를 스캔합니다."""
    open_ports_on_host = {21, 22, 80, 443, 8080}
    print(f"호스트 '{host}'의 포트 스캔 시작 ({start_port}~{end_port})...")
    for port in range(start_port, end_port + 1):
        if port in open_ports_on_host:
            print(f" Port {port}: Open")
    print("스캔 완료.")

10. 내 IP 주소 알아내기 (실제)

import socket
def get_my_local_ip():
    """외부와 통신하는 데 사용되는 로컬 IP 주소를 찾습니다."""
    s = None
    try:
        s = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
        s.connect(("8.8.8.8", 80))
        ip = s.getsockname()[0]
    except Exception as e:
        ip = "127.0.0.1"
        print(f"IP를 찾는 중 오류 발생: {e}")
    finally:
        if s:
            s.close()
    return ip
# 테스트
my_ip = get_my_local_ip()
print(f"현재 시스템에서 사용 중인 로컬 IP 주소 (추정): {my_ip}")

상세 풀이 및 설명: 이 코드는 외부 인터넷과 실제로 통신하는 '척' 함으로써, 운영체제가 어떤 네트워크 인터페이스(랜카드)를 사용하는지 알아내는 트릭을 사용합니다. socket.socket(socket.AF_INET, socket.SOCK_DGRAM): UDP 통신을 위한 소켓을 만듭니다. s.connect(("8.8.8.8", 80)): 이 코드가 핵심입니다. 8.8.8.8이라는 외부 IP로 데이터를 보내려면 어떤 로컬 IP 주소(네트워크 카드)를 출발점으로 사용해야 할지 운영체제가 결정합니다. 이 결정 과정에서 소켓 s에는 해당 로컬 IP 주소가 내부적으로 할당됩니다. s.getsockname()[0]: getsockname()은 소켓에 할당된 (IP 주소, 포트 번호) 튜플을 반환합니다. 여기서 [0] 인덱스를 통해 IP 주소 부분만 추출합니다. try...finally: finally 블록은 try문에서 오류가 발생하더라도 항상 실행됩니다. s.close()를 finally에 넣어, 어떤 상황에서든 사용한 소켓 리소스가 안전하게 닫히도록 보장합니다.
