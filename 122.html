1. 학습 목표

개념적 목표

  • 사물 인터넷 시스템의 동작 원리를 '감지-연결-처리-응용'의 4계층 아키텍처로 나누어 설명할 수 있다.
  • IoT 통신에서 MQTT 프로토콜의 역할과 발행/구독(Publish/Subscribe) 모델의 장점을 이해한다.
  • 사물(Device), 게이트웨이(Gateway), 클라우드(Cloud), 사용자 애플리케이션(Application) 간의 데이터 흐름을 설명할 수 있다.
  • 규칙 엔진(Rule Engine)이 데이터를 기반으로 자동화된 동작을 트리거하는 원리를 이해한다.
  • API(Application Programming Interface)가 플랫폼의 데이터와 서비스를 외부 애플리케이션에 제공하는 '창구' 역할을 함을 설명할 수 있다.

절차적 목표

  • Python 코드를 사용하여 MQTT의 발행/구독 모델(브로커, 발행자, 구독자)을 간단히 시뮬레이션할 수 있다.
  • 센서 데이터를 JSON 형식으로 변환(직렬화)하고, 이를 파싱하여 사용하는 과정을 코드로 구현할 수 있다.
  • 수신된 센서 데이터에 기반하여 특정 조건을 만족하면 액추에이터를 동작시키는 규칙 엔진을 구현할 수 있다.
  • 클래스를 사용하여 4계층(장치, 네트워크, 플랫폼, 앱)의 역할을 각각 모델링하고 상호작용하는 통합 시스템을 설계할 수 있다.

태도적 목표

  • 복잡한 시스템을 여러 계층으로 나누어 분석하고 이해하는 시스템적 사고(Systems Thinking) 능력을 기른다.
  • 표준화된 프로토콜(MQTT)과 데이터 형식(JSON)이 이기종 장치 간의 원활한 통신에 기여하는 가치를 인식한다.
  • 단순한 데이터의 나열이 아닌, 데이터의 '흐름'과 '처리 과정'을 통해 새로운 서비스가 창출됨을 이해하고 데이터 기반 문제 해결에 흥미를 가진다.

2. 핵심 개념 상세 설명

IoT 4계층 아키텍처 (4-Layer Architecture)

IoT 시스템의 복잡한 동작 원리를 이해하기 쉽게 나눈 4개의 논리적 계층. 데이터가 생성되어 최종적으로 사용자에게 서비스되기까지의 흐름을 보여줍니다.

  • 사물/감지 계층 (Things/Sensing Layer): 물리적 세상과 가장 가까운 계층. 센서가 환경 데이터를 수집하거나 액추에이터가 동작을 수행합니다. 데이터가 최초로 생성되는 곳입니다. (예: 스마트 워치의 심박수 센서)
  • 네트워크/연결 계층 (Network/Connectivity Layer): 감지 계층에서 생성된 데이터를 상위 계층으로 전송하고, 상위 계층의 제어 신호를 하위 계층으로 전달하는 통신 인프라. Wi-Fi, 5G, LoRaWAN 등 다양한 통신 기술과 함께, 데이터 교환을 위한 MQTT와 같은 프로토콜이 동작합니다. 이 계층에는 여러 센서 데이터를 모아 클라우드로 한 번에 전송하는 게이트웨이가 포함될 수 있습니다.
  • 플랫폼/처리 계층 (Platform/Processing Layer): IoT 시스템의 '두뇌'. 클라우드 서버에서 동작하며, 수많은 장치로부터 수집된 데이터를 저장(DB), 처리, 분석합니다. 데이터의 유효성을 검사하고, 특정 조건에 따라 장치를 제어하는 **규칙 엔진(Rule Engine)**이 동작하며, 인공지능(AI) 모델을 이용해 데이터를 분석하고 미래를 예측하기도 합니다.
  • 응용/서비스 계층 (Application/Service Layer): 처리된 데이터를 사용자에게 의미 있는 정보로 시각화하여 보여주거나, 사용자가 시스템을 제어할 수 있는 인터페이스를 제공하는 최종 단계. 스마트폰 앱, 웹 대시보드 등이 여기에 해당하며, 플랫폼의 기능을 외부에서 사용할 수 있도록 API를 제공합니다.

MQTT (Message Queuing Telemetry Transport)

IoT 환경을 위해 개발된 매우 가볍고 효율적인 메시징 프로토콜. '발행/구독(Publish/Subscribe)' 모델을 사용합니다.

  • 발행/구독 모델: 메시지를 보내는 측(발행자)과 받는 측(구독자)이 서로를 직접 알지 못하고, '브로커(Broker)'라는 중개인을 통해 통신하는 방식입니다.
  • 브로커 (Broker): 모든 메시지가 거쳐가는 중앙 우체국.
  • 주제 (Topic): 메시지의 종류를 구분하는 일종의 '채널' 또는 '게시판 이름'. (예: /home/living_room/temperature)
  • 발행 (Publish): 발행자가 특정 주제에 대해 브로커에게 메시지를 보냄.
  • 구독 (Subscribe): 구독자가 특정 주제를 브로커에게 구독 신청해 놓으면, 해당 주제로 새 메시지가 올 때마다 브로커가 전달해 줌.
  • 장점: 발행자와 구독자 간의 의존성이 없어(느슨한 결합), 수많은 장치를 유연하게 추가하고 관리하기에 용이합니다.

JSON (JavaScript Object Notation)

속성-값 쌍으로 이루어진 개방형 표준 데이터 형식. 사람이 읽고 쓰기 쉬우며 기계가 파싱하고 생성하기도 쉽습니다. IoT 환경에서 구조화된 데이터를 주고받을 때 널리 사용됩니다.

1단계: 데이터의 탄생 (감지 계층)

모든 것은 센서가 물리적 현상을 디지털 데이터로 바꾸는 것에서 시작됨을 강조. 이때 데이터는 단순한 숫자가 아니라, '언제', '어디서', '무엇을' 측정했는지 알 수 있는 구조화된 형태가 되어야 함을 설명. (JSON 형식의 유용성)

과학 예시 코드: 화성 탐사 로버 '퍼서비어런스'가 온도와 방사선량을 측정하여 JSON 형태의 데이터 패킷을 생성하는 과정을 시뮬레이션합니다.

# 상세 설명: 이 코드는 1단계 감지 계층의 역할을 보여줍니다. 
# generate_data_packet 메서드는 가상의 센서 값을 포함한 Python 딕셔너리를 생성합니다. 
# json.dumps() 함수는 이 딕셔너리를 JSON 형식의 '문자열'로 변환(직렬화)합니다.
# 이 문자열이 바로 네트워크를 통해 전송될 데이터의 실제 형태입니다. 
# 이를 통해 학생들은 데이터가 단순한 숫자가 아닌, 체계적인 구조를 가진 정보로 생성됨을 이해합니다. 
# time.time()은 타임스탬프를 찍어 데이터 생성 시점을 기록합니다.
import json
import random
import time

class PerseveranceSensor:
    def __init__(self):
        self._temperature = -63.0 # 화성의 평균 온도

    def generate_data_packet(self):
        # 화성의 환경 데이터를 측정한다고 가정
        self._temperature += random.uniform(-1, 1) # 온도는 계속 변함
        data = {
            "rover_id": "PERSEVERANCE-01",
            "timestamp": time.time(),
            "location": {"sol": 354, "site": "Jezero Crater"},
            "readings": {
                "temperature_celsius": round(self._temperature, 2),
                "radiation_mSv": 0.75 + random.uniform(-0.1, 0.1)
            }
        }
        # 딕셔너리를 JSON 문자열로 변환 (직렬화)
        return json.dumps(data, indent=2)

# 실행
rover_sensor = PerseveranceSensor()
packet_data = rover_sensor.generate_data_packet()
print("1. [감지 계층] 화성 로버가 생성한 데이터 패킷 (JSON):")
print(packet_data)

2단계: 데이터의 여정 (네트워크 계층)

생성된 데이터 패킷(JSON 문자열)을 클라우드로 전송하는 단계. 수만 개의 장치가 동시에 데이터를 보낼 때, 서버와 일일이 연결하는 것은 비효율적임을 지적하며 MQTT의 발행/구독 모델을 소개합니다. '우체국(브로커)', '잡지 이름(주제)', '기사 투고(발행)', '잡지 구독(구독)' 비유로 MQTT의 동작 원리를 설명합니다.

실생활 예시 코드: 화성 로버(발행자)가 브로커에게 데이터를 발행하고, 지구의 관제 센터(구독자)가 데이터를 수신하는 MQTT 통신을 시뮬레이션합니다.

# 상세 설명: 이 코드는 2단계 네트워크 계층의 핵심인 MQTT 통신을 보여줍니다. 
# MqttBroker 클래스는 중앙에서 메시지를 중개하는 서버 역할을 합니다.
# subscribe 메서드는 구독자의 정보를 주제별로 등록하고, publish 메서드는 특정 주제로 메시지가 들어오면 해당 주제를 구독하는 모든 구독자에게 메시지를 전달(on_message 호출)합니다. 
# 발행자(로버)와 구독자(관제 센터)는 서로를 모르고 오직 브로커와 주제를 통해 통신합니다. 이는 IoT 시스템의 유연성과 확장성을 보장하는 핵심 원리입니다.

class MqttBroker: # 가상 MQTT 브로커
    def __init__(self):
        self.subscribers = {} # { "주제": [구독자1, 구독자2...] }

    def subscribe(self, topic, subscriber):
        if topic not in self.subscribers:
            self.subscribers[topic] = []
        self.subscribers[topic].append(subscriber)
        print(f"2a. [네트워크 계층] '{subscriber.name}'이(가) '{topic}' 주제를 구독 시작.")

    def publish (self, topic, message):
        print(f"\n2b. [네트워크 계층] 발행자가 '{topic}' 주제로 메시지 발행!")
        if topic in self.subscribers:
            for subscriber in self.subscribers[topic]:
                subscriber.on_message(topic, message) # 구독자에게 메시지 전달

class Subscriber: # 관제 센터 역할
    def __init__(self, name): self.name = name
    def on_message(self, topic, message):
        print(f" -> '{self.name}' 수신: {message[:50]}...")

# 시스템 설정
broker = MqttBroker()
mission_control = Subscriber(name="NASA 관제 센터")
broker.subscribe("/mars/perseverance/env_data", mission_control)

# 발행
broker.publish("/mars/perseverance/env_data", packet_data)

3단계: 데이터의 해석 (플랫폼 계층)

브로커를 통해 클라우드에 도착한 데이터가 '의미 있는 행동'으로 이어지는 단계. 이 계층의 핵심 역할은 '규칙 기반의 자동화'임을 설명. (만약 ~하면, ~하라)

과학 예시 코드: 플랫폼에 '규칙 엔진' 구독자를 추가합니다. 이 구독자는 화성 로버의 온도가 너무 낮아지면 히터를 켜라는 명령을 생성합니다.

# 상세 설명: 이 코드는 3단계 플랫폼 계층의 '지능'을 보여줍니다.
# RuleEngine은 MQTT 브로커로부터 환경 데이터를 구독합니다. 
# on_message 메서드 안에서 json.loads() 함수를 이용해 수신된 JSON 문자열을 다시 Python 딕셔너리로 변환(역직렬화)하여 데이터에 접근합니다. 
# if temp < -63: 라는 조건문이 바로 '규칙'에 해당합니다. 
# 조건이 만족되면, 규칙 엔진은 이제 '발행자'가 되어 히터를 제어하는 새로운 주제(/commands)로 제어 명령을 발행합니다.

class RuleEngine(Subscriber): # 규칙 엔진도 하나의 구독자
    def __init__(self, name, broker):
        super().__init__(name)
        self.broker = broker

    def on_message(self, topic, message):
        print("\n3. [플랫폼 계층] 규칙 엔진이 데이터 수신 및 분석...")
        data = json.loads(message) # JSON 문자열을 딕셔너리로 변환 (역직렬화)
        temp = data["readings"]["temperature_celsius"]
        
        if temp < -63.0:
            print(f" -> 분석 결과: 온도({temp}°C)가 위험 수준! 히터 가동 필요.")
            # 분석 결과에 따라 새로운 명령을 다른 주제로 발행!
            command = {"command": "TURN_ON_HEATER", "power": "HIGH"}
            self.broker.publish("/mars/perseverance/commands", json.dumps(command))

# 시스템에 규칙 엔진 추가
rule_engine = RuleEngine(name="자동화 규칙 엔진", broker=broker)
broker.subscribe("/mars/perseverance/env_data", rule_engine) # 환경 데이터를 구독

# 위험한 온도 데이터 생성 및 발행
dangerous_packet = rover_sensor.generate_data_packet().replace(str(round(rover_sensor._temperature, 2)), "-65.0")
broker.publish("/mars/perseverance/env_data", dangerous_packet)

4단계: 결과의 활용 (응용 계층)

처리된 데이터나 시스템의 상태를 사용자가 보거나 제어하는 최종 단계. 스마트폰 앱이나 웹 대시보드가 이 계층에 해당함을 설명.

실생활 예시 코드: 사용자 앱(대시보드) 역할을 하는 구독자를 추가하여, 로버의 상태와 히터 가동 명령을 모두 화면에 출력합니다.

# 상세 설명: 이 코드는 4단계 응용 계층의 역할을 보여줍니다.
# UserApplication 클래스는 사용자에게 보여질 정보와 관련된 모든 주제(환경 데이터, 제어 명령)를 구독합니다.
# 그리고 각 주제에 맞는 메시지가 도착하면, 그에 따라 화면을 업데이트(여기서는 print문)합니다.
# 이를 통해 사용자는 IoT 시스템의 현재 상태를 실시간으로 모니터링하고, 시스템이 내린 결정(히터 켜기 등)을 확인할 수 있습니다. 이것이 바로 IoT가 사용자에게 '서비스'와 '가치'를 제공하는 최종적인 모습입니다.

class UserApplication(Subscriber): # 사용자 앱/대시보드
    def on_message(self, topic, message):
        print(f"\n4. [응용 계층] 사용자 앱 화면 업데이트!")
        if topic == "/mars/perseverance/env_data":
            data = json.loads(message)
            temp = data["readings"]["temperature_celsius"]
            print(f" -> 대시보드: 현재 로버 온도 = {temp:.2f}°C")
        elif topic == "/mars/perseverance/commands":
            command = json.loads(message)["command"]
            print(f" -> 알림: '{command}' 명령이 로버에 전송되었습니다!")

# 시스템에 사용자 앱 추가
app = UserApplication(name="지구 관제사 앱")
broker.subscribe("/mars/perseverance/env_data", app) # 환경 데이터 구독
broker.subscribe("/mars/perseverance/commands", app) # 제어 명령도 구독

# 이전 단계의 위험 데이터 발행을 다시 실행하면 앱에도 표시됨
broker.publish("/mars/perseverance/env_data", dangerous_packet)

4. 진단평가 (OX 퀴즈)

1. 사물 인터넷의 감지 계층은 수집된 데이터를 분석하고 처리하는 역할을 한다. (X)

상세 풀이: 감지 계층은 센서를 통해 물리적 환경의 데이터를 '수집'하고 생성하는 역할을 합니다. 데이터의 분석과 처리는 '플랫폼/처리 계층'에서 이루어집니다.

2. MQTT 프로토콜에서 발행자(Publisher)는 구독자(Subscriber)의 IP 주소를 직접 알아야 통신할 수 있다. (X)

상세 풀이: MQTT는 '브로커'를 중간에 두는 발행/구독 모델을 사용합니다. 발행자와 구독자는 서로의 존재를 모르며, 오직 브로커와 '주제(Topic)'를 통해서만 통신합니다. 이것이 MQTT의 핵심적인 장점입니다.

3. JSON은 IoT 환경에서 구조화된 데이터를 교환하기 위해 널리 사용되는 데이터 형식이다. (O)

상세 풀이: JSON은 사람이 읽기 쉽고 기계가 처리하기 쉬운 형식으로, 여러 속성을 가진 센서 데이터(예: 온도, 습도, 시간)를 하나의 구조로 묶어 표현하는 데 매우 유용합니다.

4. IoT의 응용/서비스 계층은 사용자가 시스템의 상태를 확인하거나 제어하는 스마트폰 앱, 웹 대시보드 등을 포함한다. (O)

상세 풀이: 응용/서비스 계층은 처리된 데이터를 사용자에게 가치 있는 정보로 보여주는 최종 단계로, 사용자와 시스템 간의 상호작용이 일어나는 인터페이스 역할을 합니다.

5. 기본 게이트웨이(Gateway)는 일반적으로 플랫폼 계층에 속하는 요소이다. (X)

상세 풀이: 게이트웨이는 여러 센서 장치(감지 계층)의 데이터를 모아 상위 네트워크(클라우드 플랫폼)로 전송하는 역할을 하므로, '네트워크/연결 계층'에 속하는 장비입니다.

6. 규칙 엔진(Rule Engine)은 센서 데이터가 특정 조건을 만족할 때 자동으로 정해진 동작을 실행시키는 역할을 한다. (O)

상세 풀이: 규칙 엔진은 'IF-THEN' 형식의 규칙들을 기반으로 동작합니다. 예를 들어, '만약(IF) 온도가 30도 이상이면, 에어컨을 켜라(THEN)'와 같은 자동화 로직을 수행합니다.

7. API(Application Programming Interface)는 센서가 데이터를 수집하는 방식을 정의하는 규칙이다. (X)

상세 풀이: API는 플랫폼 계층이 자신의 데이터나 기능을 외부 응용 프로그램(앱, 다른 서비스 등)이 사용할 수 있도록 제공하는 '인터페이스' 또는 '메뉴판'입니다. 센서의 데이터 수집 방식과는 거리가 멉니다.

8. MQTT의 '주제(Topic)'는 메시지의 내용을 의미한다. (X)

상세 풀이: '주제(Topic)'는 메시지의 내용(payload)이 아니라, 메시지를 분류하는 '채널' 또는 '주소'의 역할을 합니다. 구독자는 이 주제를 기준으로 원하는 메시지를 선택하여 받습니다.

9. IoT 시스템의 모든 데이터 처리는 반드시 클라우드에서만 이루어져야 한다. (X)

상세 풀이: 최근에는 빠른 응답이 필요하거나 인터넷 연결이 불안정한 경우를 대비하여, 데이터를 클라우드까지 보내지 않고 게이트웨이나 장치 자체에서 일부를 처리하는 '에지 컴퓨팅(Edge Computing)' 또는 '포그 컴퓨팅(Fog Computing)' 기술도 중요해지고 있습니다.

10. IoT 아키텍처에서 데이터는 보통 '응용 계층 → 플랫폼 계층 → 네트워크 계층 → 감지 계층' 순서로 흐른다. (X)

상세 풀이: 데이터의 흐름은 그 반대입니다. 데이터는 '감지 계층'에서 생성되어 '네트워크 계층'을 통해 '플랫폼 계층'으로 전송되어 처리된 후, 최종적으로 '응용 계층'에서 사용자에게 보여집니다.

  2. 발행자와 구독자 간의 1:1 직접 통신을 기반으로 한다.
  3. 동영상 스트리밍과 같은 대용량 데이터 전송에 최적화되어 있다.

5. 다음 중 MQTT의 '주제(Topic)'에 대한 설명으로 가장 올바른 것은?

상세 풀이: JSON은 키:값 쌍의 단순한 구조를 가진 텍스트 형식이라 개발자가 디버깅하기 쉽고, 대부분의 프로그래밍 언어에서 쉽게 파싱(해석)할 수 있는 라이브러리를 지원하기 때문에 이기종 장치 간 데이터 교환에 널리 사용됩니다.

}

# 2. 딕셔너리를 JSON 형식의 문자열로 변환 (직렬화)
json_string = json.dumps(student_info, indent=2, ensure_ascii=False)

2-5. 통합 MQTT 발행/구독/브로커 시뮬레이션

Publisher, Subscriber, Broker 클래스를 모두 사용하여, 온도 센서(발행자)가 1초마다 온도를 "/iot/home/temp" 주제로 발행하고, 모니터(구독자)는 이 온도를 받아 화면에 출력하는 전체 시스템을 코드로 구현하세요.

import time
import random

# 4. Broker 클래스 정의
class Broker:
    def __init__(self):
        print("브로커가 생성되었습니다.")
        self.subscribers = {}

    def add_subscriber(self, topic, subscriber):
        if topic not in self.subscribers: self.subscribers[topic] = []
        self.subscribers[topic].append(subscriber)
        print(f"'{subscriber.name}'이(가) '{topic}' 주제를 구독합니다.")

    def receive_publish(self, topic, message):
        print(f"\n브로커: '{topic}' 주제로 메시지 수신!")
        if topic in self.subscribers:
            for sub in self.subscribers[topic]:
                sub.on_message(topic, message)

# 2. Publisher 클래스 정의
class Publisher:
    def __init__(self, name): self.name = name
    def publish(self, broker, topic, message):
        print(f"발행자({self.name}): '{topic}' 주제로 '{message}' 발행 시도.")
        broker.receive_publish(topic, message)

# 3. Subscriber 클래스 정의
class Subscriber:
    def __init__(self, name): self.name = name
    def subscribe(self, broker, topic): broker.add_subscriber(topic, self)
    def on_message(self, topic, message):
        print(f" -> 구독자({self.name}): 메시지 도착! (주제: {topic}, 내용: {message})")

# 5. 통합 시스템 실행
print("\n--- MQTT 시스템 시뮬레이션 시작 ---")
broker = Broker()
temp_sensor = Publisher(name="온도 센서")
monitor_app = Subscriber(name="모니터링 앱")

monitor_app.subscribe(broker, "/home/living_room/temp")

# 1초마다 데이터 발행 시뮬레이션
for _ in range(3):
    temp_reading = f"{random.uniform(20, 25):.1f}°C"
    temp_sensor.publish(broker, "/home/living_room/temp", temp_reading)
    # 다른 주제로 발행된 메시지는 수신 안 됨
    temp_sensor.publish(broker, "/home/kitchen/humidity", "60%")
    time.sleep(1)

상세 풀이 및 설명: 이 코드는 MQTT 통신의 세 주체(Broker, Publisher, Subscriber)가 어떻게 상호작용하는지 보여주는 통합 예제입니다. 실행 흐름: 1. 각 객체를 생성합니다. 2. `monitor_app`이 `broker`에 `/home/living_room/temp` 주제를 `subscribe`합니다. 3. `temp_sensor`가 같은 주제로 메시지를 `publish`하면, `broker`는 이 주제를 구독하는 `monitor_app`을 찾아 `on_message` 메서드를 호출해줍니다. 다른 주제(`.../humidity`)로 발행된 메시지는 `monitor_app`이 구독하지 않았으므로 전달되지 않습니다. 이를 통해 발행자와 구독자가 서로를 알 필요 없이 오직 브로커와 주제를 통해 통신하는 MQTT의 핵심 원리를 이해할 수 있습니다.

6. 간단한 규칙 엔진 만들기

온도 메시지({"temp": 28})를 받는 RuleEngine 구독자를 만드세요. 만약 수신한 온도가 27도보다 높으면, "경고: 온도가 너무 높습니다! 에어컨을 켜세요." 라는 메시지를 "/iot/home/alert" 주제로 다시 발행(publish)하도록 작성하세요.

import json
# (이전 예제의 Broker, Publisher, Subscriber 클래스가 필요합니다)

class RuleEngine(Subscriber):
    def __init__(self, name, broker):
        super().__init__(name)
        self.broker = broker # 다른 주제로 발행하기 위해 브로커를 알아야 함

    def on_message(self, topic, message):
        print(f"\n규칙 엔진: '{topic}' 데이터 수신: {message}")
        data = json.loads(message)
        temp = data.get("temp")

        if temp is not None and temp > 27:
            print(" -> 규칙 발동: 온도가 27도 초과! 경고 발생!")
            alert_message = f"경고: 온도가 위험 수준({temp}°C)입니다."
            # 새로운 주제로 경고 메시지를 다시 발행
            self.broker.receive_publish("/home/living_room/alert", alert_message)

# 테스트
broker = Broker()
temp_sensor = Publisher(name="온도 센서")
rule_engine = RuleEngine(name="자동화 규칙 엔진", broker=broker)
alert_monitor = Subscriber(name="경고 모니터") # 경고 메시지를 받을 구독자

# 구독 설정
rule_engine.subscribe(broker, "/home/living_room/temp_json")
alert_monitor.subscribe(broker, "/home/living_room/alert")

# 데이터 발행
# 정상 온도
temp_data_normal = json.dumps({"temp": 25})
temp_sensor.publish(broker, "/home/living_room/temp_json", temp_data_normal)
# 위험 온도
temp_data_high = json.dumps({"temp": 28})
temp_sensor.publish(broker, "/home/living_room/temp_json", temp_data_high)

상세 풀이 및 설명: RuleEngine은 Subscriber이기도 합니다. 즉, 다른 장치와 마찬가지로 특정 주제의 메시지를 구독하여 받습니다. on_message 메서드에서 수신한 JSON 메시지를 json.loads로 파싱하여 내부 데이터(temp)에 접근합니다. `if temp > 27:` 부분이 바로 '규칙'입니다. 규칙이 만족되면, RuleEngine은 이제 '발행자'의 역할을 수행합니다. `self.broker.receive_publish`를 호출하여 새로운 주제(/home/living_room/alert)로 새로운 메시지(경고)를 발행합니다. 이 시스템에서는 `alert_monitor`라는 또 다른 구독자가 이 경고 메시지를 수신하게 됩니다. 이는 IoT에서 데이터가 한 방향으로만 흐르는 것이 아니라, 처리 결과에 따라 다시 새로운 데이터가 되어 다른 시스템으로 전파되는 연쇄 반응을 보여줍니다.

8. 와일드카드 구독자 (개념 설명 코드)

MQTT 주제는 와일드카드(+, #)를 지원합니다. Broker 클래스를 수정하여, "/home/+/temp"(모든 방의 온도)나 "/home/#"(home 아래의 모든 주제)와 같은 와일드카드 구독을 처리할 수 있도록 만들어 보세요.

# (이전 예제의 Broker, Publisher, Subscriber 클래스가 필요합니다)

class BrokerWithWildcard(Broker):
    def receive_publish(self, topic, message):
        print(f"\n브로커: '{topic}' 주제로 메시지 수신!")
        # 정확히 일치하는 구독자에게 전달
        if topic in self.subscribers:
            for sub in self.subscribers[topic]:
                sub.on_message(topic, message)

        # 와일드카드 구독자에게도 전달
        for sub_topic, subscribers in self.subscribers.items():
            # 간단한 와일드카드 매칭 로직 ('#'만 구현)
            if sub_topic.endswith('/#'):
                root_topic = sub_topic[:-2]
                if topic.startswith(root_topic):
                    for sub in subscribers:
                        sub.on_message(topic, message)

# 테스트
broker = BrokerWithWildcard()
all_home_monitor = Subscriber(name="집 전체 모니터")
living_room_monitor = Subscriber(name="거실 모니터")

# 와일드카드 구독
all_home_monitor.subscribe(broker, "/home/#")
# 일반 구독
living_room_monitor.subscribe(broker, "/home/living_room/temp")

p = Publisher("센서")
p.publish(broker, "/home/living_room/temp", "25°C") # 둘 다 수신
p.publish(broker, "/home/kitchen/gas", "OK") # '집 전체 모니터'만 수신

상세 풀이 및 설명: 이 예제는 Broker의 `receive_publish`가 어떻게 와일드카드를 처리할 수 있는지 그 개념을 보여줍니다. `#`(multi-level wildcard): `/home/#`는 `/home/living_room/temp`, `/home/kitchen/gas` 등 `/home/`으로 시작하는 모든 주제에 매칭됩니다. `+`(single-level wildcard): `/home/+/temp`는 `/home/living_room/temp`, `/home/bedroom/temp`에는 매칭되지만, `/home/living_room/light/temp`에는 매칭되지 않습니다. 이러한 와일드카드 구독 기능은 하나의 구독자가 여러 센서의 데이터를 종류별로 한 번에 받아보는 등 유연한 시스템을 구성하는 데 매우 유용합니다.

9. 게이트웨이 시뮬레이션

여러 개의 SimpleSensor(ID만 가짐) 객체를 만드세요. Gateway 클래스를 만들어 이 센서들을 등록하게 하세요. Gateway의 collect_and_publish(broker) 메서드는 등록된 모든 센서의 데이터를 한 번에 모아 [{"id": "s1", "value": 10}, {"id": "s2", "value": 20}] 과 같은 JSON 배열 형태로 만들어 "/gateway/all_data" 주제로 발행하도록 작성하세요.

import json
import random
# (간단한 Broker와 Subscriber 클래스가 있다고 가정)
class Broker:
    def receive_publish(self, t, m): print(f"Broker: {t} - {m}")
class Subscriber:
    def __init__(self,n): self.name=n
    def subscribe(self,b,t): print(f"{self.name} subscribes to {t}")

class SimpleSensor:
    def __init__(self, device_id): self.id = device_id
    def read_value(self): return random.randint(1, 100)

class Gateway:
    def __init__(self, gateway_id):
        self.id = gateway_id
        self.sensors = []
    def register_sensor(self, sensor):
        self.sensors.append(sensor)
        print(f"게이트웨이({self.id}): 센서({sensor.id}) 등록 완료.")
    def collect_and_publish(self, broker, topic):
        print(f"\n게이트웨이({self.id}): 연결된 센서들의 데이터 수집 시작...")
        all_data = [{"id": sensor.id, "value": sensor.read_value()} for sensor in self.sensors]
        print("게이트웨이: 데이터 취합 완료. 클라우드로 발행합니다.")
        broker.receive_publish(topic, json.dumps(all_data))

# 테스트
broker = Broker()
cloud_server = Subscriber("클라우드 서버")
cloud_server.subscribe(broker, "/gateway/01/all_data")

gateway = Gateway("GW-01")
gateway.register_sensor(SimpleSensor("Sensor-A"))
gateway.register_sensor(SimpleSensor("Sensor-B"))

gateway.collect_and_publish(broker, "/gateway/01/all_data")

상세 풀이 및 설명: 이 시나리오는 모든 센서가 직접 인터넷에 연결되는 것이 아니라, 근거리 통신으로 '게이트웨이'에 연결되는 상황을 모델링합니다. `collect_and_publish` 메서드가 핵심입니다. 이 메서드는 자신에게 연결된 모든 센서(`self.sensors`)를 순회하며 `read_value()`를 호출하여 데이터를 수집합니다. 수집된 개별 데이터를 하나의 큰 리스트(`all_data`)에 담습니다. 이 리스트를 `json.dumps`를 이용해 JSON 배열 문자열로 만들어, 클라우드(브로커)로 한 번에 전송합니다. 이는 개별 센서들이 직접 인터넷 통신을 하지 않아도 되므로, 센서의 배터리 소모를 줄이고 시스템을 효율적으로 만드는 데 기여합니다.

10. 디지털 트윈 개념 시뮬레이션

Motor 클래스를 만드세요. 실제 모터의 상태(rpm, temperature)와, 클라우드에 복제된 가상 모터의 상태를 각각 속성으로 가지게 하세요. update_physical_state() 메서드는 실제 모터의 상태를 랜덤하게 변경하고, sync_to_cloud() 메서드는 실제 상태를 가상 상태로 복사하며 "클라우드와 동기화 완료" 메시지를 출력하도록 작성하세요.

import random
import time

class Motor:
    """물리적 모터와 그 상태를 복제한 디지털 트윈을 시뮬레이션하는 클래스"""
    def __init__(self, motor_id):
        self.id = motor_id
        # 1. 물리적 세상의 실제 모터 상태
        self.physical_rpm = 0
        self.physical_temp = 20.0
        # 2. 클라우드 상의 가상 모터 (디지털 트윈) 상태
        self.digital_twin_rpm = 0
        self.digital_twin_temp = 20.0
        print(f"모터({self.id}) 생성. 물리/가상 상태 초기화 완료.")

    def update_physical_state(self):
        self.physical_rpm += random.randint(-10, 50)
        self.physical_temp += random.uniform(-1.0, 3.0)
        if self.physical_rpm < 0: self.physical_rpm = 0
        print(f"\n물리적 상태 변경: RPM={self.physical_rpm}, 온도={self.physical_temp:.1f}°C")

    def sync_to_cloud(self):
        print("동기화 시작: 물리적 상태 -> 클라우드 디지털 트윈")
        self.digital_twin_rpm = self.physical_rpm
        self.digital_twin_temp = self.physical_temp
        print("동기화 완료!")
    
    def display_states(self):
        print(f" - 물리적 상태: RPM={self.physical_rpm}, 온도={self.physical_temp:.1f}°C")
        print(f" - 디지털 트윈: RPM={self.digital_twin_rpm}, 온도={self.digital_twin_temp:.1f}°C")

# 테스트
factory_motor = Motor("Motor-007")
for _ in range(2):
    factory_motor.update_physical_state()
    factory_motor.display_states()
    time.sleep(1)
    factory_motor.sync_to_cloud()
    print("동기화 후 상태:")
    factory_motor.display_states()

상세 풀이 및 설명: 디지털 트윈이란 현실 세계의 물리적 자산을 가상 공간에 똑같이 복제하는 기술을 의미합니다. Motor 클래스는 이를 단순화하여, `physical_` 접두사가 붙은 속성(실제 모터의 상태)과 `digital_twin_` 접두사가 붙은 속성(클라우드에 저장된 가상 모터의 상태)을 모두 가집니다. `update_physical_state`는 공장에서 모터가 실제로 돌아가면서 RPM과 온도가 변하는 물리적 현상을 시뮬레이션합니다. `sync_to_cloud`는 모터에 부착된 센서가 현재 상태를 읽어 클라우드로 전송하는 '동기화' 과정을 추상화한 것입니다. 이 메서드가 호출되면, 물리적 상태 값이 디지털 트윈 상태 값으로 복사됩니다. IoT 플랫폼은 이 디지털 트윈의 데이터를 보고 모터의 상태를 원격으로 모니터링하거나, 고장 예측 AI 모델을 돌리는 등 다양한 분석 및 제어 작업을 수행할 수 있습니다.
